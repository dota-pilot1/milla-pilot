A · 구조 이해

Q1 카탈로그에서 담고 → 후원 목록 되고 → 후원자가 결제, 이 절차가 맞나?

큰 줄기는 맞다. 다만 중간에 운영팀 검수·프로젝트 발행이라는 관문이 하나 있고, 마지막 단계는 "기부금 납부"가 아니라 물품 결제다. 
카탈로그(상품정보)
  → 시설이 장바구니에 담아 제출        [돈 안 움직임]
  → 운영팀 검수 (승인/보류/반려)        [돈 안 움직임]
  → "프로젝트"로 묶어 발행 + 공개 예약   [돈 안 움직임]
  ─────────────────────────────────────
  → 기부자가 물품 선택 → 학교매점 결제   [협력사로 입금]
  → 협력사가 소싱사이트에 발주 → 시설 배송
  → 시설 수령확인 → 구매영수증 전송 → 기부영수증 발급

점선 위쪽이 Phase A(지금 착수 가능), 아래쪽이 Phase C(세무·협력사 관문 통과 후).

Q2 장바구니가 두 개라는 게 무슨 말인가?

기획서에 "장바구니"가 두 번 나오는데 서로 다른 것이다. 이름이 같아서 도메인 설계 때 가장 많이 꼬이는 지점이다.

구분 담는 사람 의미 결제
시설 장바구니 시설 담당자 "이 물품들이 필요합니다" 요청서를 만드는 도구. 제출하면 검수로 넘어감 없음
학교매점 장바구니 기부자 발행된 프로젝트에서 후원할 물품을 고르는 것 있음
도메인 이름을 처음부터 분리한다 — 예: ItemRequest(시설 제출건) vs StoreCart(기부자 결제용). 둘 다 Cart로 부르면 반드시 사고가 난다.

Q3 기부자는 돈을 내는 건가, 물건을 사는 건가?

산다. 기부금 송금 방식은 없어진다. 이게 이 기획의 전체 요점이다.
지금 (1차 MVP) 학교매점 구조
기부자 행위 금액을 후원 물품을 결제
돈이 가는 곳 시설 후원금 전용계좌 협력사 계좌
시설이 받는 것 돈 (직접 구매) 물건
기부자 확인 시점 사후 알림장 결제 시점

화면 문구도 "얼마를 후원하시겠습니까"가 아니라 "이 아이의 신발을 채워주세요" 형태가 된다.

Q4 그럼 시설 후원금 전용계좌는 이제 안 쓰나?

이 구조에서는 거치지 않는다. 기부자 결제가 협력사로 바로 들어가고 시설은 물건만 받는다.

이게 기획 6-1의 핵심 쟁점이다. 시설이 물품 구매를 협력사에 적법하게 위임한 것으로 볼지, 아니면 기부자가 물건을 사서 시설로 보내는 다른 성격의 거래로 볼지가 아직 정해지지 않았다. 어느 쪽으로 정하느냐에 따라 주문·결제 도메인의 당사자 모델링이 통째로 달라진다. 전문가 사전검토 전에는 결제 코드를 쓰지 않는다.

Q5 학교 플랫폼은 돈을 한 푼도 안 만지나?

그렇다. 플랫폼 법인은 검수·기록·전달 확인만 한다. 개발상 의미는 분명하다.

플랫폼 서버 도메인에 잔액·예치금·출금·정산 개념을 만들면 안 된다. 결제 결과를 참조하고 기록할 뿐이다.

Q6 협력사는 왜 굳이 따로 만드나?

학교 플랫폼 법인이 직접 상품을 팔면 커머스 사업자가 되어버린다. 통신판매업 신고, 전자상거래법상 판매자 책임(반품·교환·표시광고), 재고 손실 부담이 전부 플랫폼 법인에 붙는다. 이걸 분리하려고 별도 사업자를 세우는 것이다.

단, 협력사 설립 방식(신규 법인 vs 기존 커머스 사업자와 제휴)은 미확정 — 기획 9장 결정 항목.

B · 카탈로그와 상품

Q7 카탈로그 상품은 누가 등록하나?

운영팀이 B2B 위탁 소싱사이트에서 끌어와 등록한다. 시설이나 기부자가 임의로 상품을 추가할 수 없는 폐쇄형이다.

취급 범위는 아동·청소년에게 꼭 필요한 것으로 제한한다.

대분류 예시
의류 겨울 패딩, 내복, 계절별 상하의
잡화 운동화, 실내화, 책가방
절대 올리지 않는 것 — 사치품·고가 브랜드, 현금성 상품권, 위험물, 개인정보를 요구하는 상품. 검수 단계에서 거르는 게 아니라 카탈로그 등록 단계에서 원천 차단하는 게 1차 방어선이다.

Q8 재고는 누가 갖고 있나?

아무도 떠안지 않는다. 소싱사이트가 물류·배송을 처리하고, 학교매점은 판매 페이지와 결제만 갖는다. 발주 시점에만 원가로 연동된다.

개발상 의미 — 우리 DB에 재고 수량을 소유하지 않는다. 소싱사이트 조회 결과를 캐시로만 취급하고, 품절 옵션은 프로젝트 등록 시점에 자동으로 걸러낸다. 소싱사이트가 아직 미정이므로 어댑터 인터페이스 + 수동입력 구현체로 먼저 만들고, 선정 후 실제 구현체를 갈아끼운다.

C · 제출과 검수

Q9 시설이 담은 게 바로 후원 목록이 되나?

아니다. 검수를 통과하고 프로젝트로 묶인 것만 노출된다.
제출 → [검수 대기중] → 운영팀 상세 검수
                          ├ 승인 → 프로젝트로 묶기 → 공개 예약 → 노출
                          ├ 보류
                          └ 반려 → 사유가 시설에 자동 전달

검수에서 보는 것: 품목·수량·가격 적정성, 중복 여부, 금지 품목 여부.

Q10 "프로젝트"는 정확히 무엇인가?

승인된 품목들을 하나의 이야기로 묶은 단위다. 예: "OO보육원 아이 10명 겨울 패딩 프로젝트". 프로젝트에는 노출 시작일·마감일이 붙고, 홈 화면 "이번 프로젝트" 영역에 자동 노출된다.

현재 코드에 없는 개념이다. 지금은 DonationItem이 Facility 직속이라 Facility → Item 2계층인데, 이걸 Facility → Project → Item 3계층으로 바꾸는 마이그레이션이 필요하다. 기존 데이터 구조를 건드리는 일이라 뒤로 미룰수록 비싸진다.

Q11 마진율은 왜 비공개인가? 그럼 어떻게 안내하나?

학교매점은 소싱 원가에 최소한의 마진만 붙이고, 그 마진을 재기부 재원 + 운영비로 쓴다. 정확한 비율·금액은 대외 공개하지 않고, 사용 방향과 "합리적인 가격으로 운영한다"는 원칙만 안내한다.

개발상 의미 — 원가·마진 필드는 DB에 존재하되, 공개 API 응답 DTO에서 원천 배제한다. 학급일지·완결 리포트에도 마진 수치가 새어나가면 안 된다. 설계 단계에서 못박아야 나중에 실수로 노출되지 않는다.

D · 결제와 영수증

Q12 구매하면 기부영수증이 자동으로 나오나?

아니다. 그리고 그렇게 안내해서도 안 된다. 기획서가 명시적으로 금지한 문구다.

자동으로 발행되는 것은 협력사의 구매영수증이다. 그게 시설로 전달되면, 시설이 확인 절차를 거쳐 기부영수증을 발급한다. 발급 주체는 끝까지 시설이다.

달라지는 건 시설의 업무가 "직접 대조해서 만드는 것" → "전달받은 자료를 확인하고 발급하는 것"으로 단순해진다는 점이지, 발급이 없어지는 게 아니다.

시스템은 시설의 발급 행위를 대행하지 않는다. 영수증 상태는 시설의 액션으로만 전진한다. 또 시설별로 발급 자격이 다를 수 있으므로 Facility에 발급 자격 플래그가 필요하고, 자격 없는 시설은 UI 안내가 달라야 한다.

Q13 영수증 4단계는 왜 필요한가?

기부자가 "내 영수증이 어디쯤 갔는지" 불안해하지 않도록, 그리고 Q12의 오해를 구조적으로 막기 위해서다. 마이페이지에서 직접 추적 가능해야 한다. 

발행됨  →  시설 전송됨  →  시설 확인 중  →  기부영수증 발급됨
 (협력사)     (협력사→시설)      (시설)          (시설→기부자)

단계가 눈에 보이면 "시설의 확인 절차가 남아 있다"가 말이 아니라 화면으로 전달된다.

Q14 결제 화면이 협력사로 넘어가면 이탈하지 않나?

기획서가 직접 지적한 리스크다. "내가 학교를 벗어나 다른 데서 결제하는 건가?"라는 의문이 들면 결제 이탈이 생긴다. 완화책 4가지:

  1. 학교매점 화면의 디자인 톤(색·서체·레이아웃)을 학교 플랫폼과 통일
  2. 결제 화면 상단에 "학교 플랫폼과 함께하는 협력 매점입니다" 안내 노출
  3. 결제 완료 후 지체 없이 학교 플랫폼 화면으로 복귀 → "참여 확정" 표시
  4. 마이페이지에서 영수증 4단계 추적 제공

Q15 목표 미달이거나 결제 후 품절이면 어떻게 되나?

기획서에 명시되지 않은 공백이다. 미확정

1차 MVP는 "돈이 모이고 목표 달성 후 구매"라 미달 처리가 단순했다. 학교매점은 결제가 곧 발주라 성격이 다르다. 최소한 아래는 정해야 한다.

  • 프로젝트 마감일까지 일부 품목만 팔린 경우 — 그대로 발주? 시설과 재협의?
  • 결제 직후 소싱사이트 품절이 확인된 경우 — 환불? 대체 상품? 누가 결정?
  • 환불이 발생하면 이미 발행된 구매영수증과 기부영수증은 어떻게 되나

품절은 "프로젝트 등록 시점 실시간 재고 연동"으로 예방은 하지만, 결제와 발주 사이의 시차는 남는다. 이건 Phase C 설계 때 반드시 답이 있어야 한다.

E · 개발 관점

Q16 기존 통합구매 기능은 버리는 건가?

기능은 남지만 의미가 뒤집힌다. 현재 PurchaseOrder는 이렇게 되어 있다.

  • 목표 달성 후 운영자가 외부에서 한 번에 사고 결과를 기록
  • donation_item_id UNIQUE — 물품당 1건으로 중복구매 원천 차단

학교매점 구조에서는 기부자 결제 건별로 소싱사이트에 발주가 나간다.

따라서 "물품당 1건" UNIQUE 제약이 성립하지 않는다. 스키마 변경이 필요하고, "통합구매"라는 화면·메뉴 이름도 재검토 대상이다.

Q17 기존 DB와 1차 MVP 데이터는 어떻게 되나?

현재 처리
Facility 유지 + 담당자 계정 매핑·발급자격 플래그 추가
DonationItem 자유입력 → 카탈로그 SKU 참조로 재정의, 상위에 Project 삽입
Contribution status에 PAID·REFUNDED 추가 (관문 통과 후)
PurchaseOrder UNIQUE 해제 + 발주 단위 재설계
menu·role·permission 그대로 재사용. 학교매점 메뉴와 시설담당자 권한만 추가
notification 그대로 재사용 (제출·반려·검수완료·수령확인 알림)

Q18 지금 당장 코딩 시작해도 되나?

절반은 된다. 돈과 영수증을 건드리지 않는 부분은 관문과 무관하게 착수 가능하다.
구분 내용
착수 가능 카탈로그 도메인 · 시설담당자 권한 · 장바구니/제출 · 운영팀 검수 화면 · Project 마이그레이션 · 기부자 프로젝트 상세(장바구니 담기까지) Phase A
착수 금지 주문·결제 · 구매영수증 4단계 · 인웹 결제 전환 UX · 소싱사이트 실발주 · PurchaseOrder 재설계 Phase C

기획 10장은 "협력사 확정 → 전문가 검토 → 개발" 순서로 적혀 있지만, 실제로는 제출·검수·발행 라인이 관문과 독립적이라 병행이 가능하다. 반대로 결제·영수증을 관문 전에 만들면 구조가 바뀌면서 전부 버리게 된다.

권장 착수 순서는 Project 마이그레이션(A-5) 먼저. 기존 데이터 구조를 바꾸는 일이라 다른 걸 얹은 뒤에 하면 비용이 커진다.

Q19 아직 안 정해진 게 뭔가?

Q20 관리앱(Tauri)과 웹은 어떻게 나뉘나?

기존 원칙 그대로다 — 관리·CRUD는 Tauri, 후원자 대면 출력은 웹.

화면 위치 근거
카탈로그 상품 등록·관리 Tauri 운영팀 전용 CRUD
운영팀 검수·프로젝트 발행 Tauri 운영팀 전용 관리 업무
시설 담당자 제출 화면 검토 외부 시설 담당자가 쓰는 화면 — 앱 설치를 요구하기 어려우므로 웹이 유력
기부자 프로젝트·결제·영수증 추적 웹 후원자 대면

시설 제출 화면의 배치는 별도 결정이 필요하다. 기획 8장이 "협력교실 담당자가 UX에 익숙하지 않아 제출을 포기"를 리스크로 꼽은 만큼, 진입 장벽이 낮은 쪽을 골라야 한다.
