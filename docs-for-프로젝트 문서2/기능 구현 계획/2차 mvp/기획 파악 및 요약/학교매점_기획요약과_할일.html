2차 MVP · 기획 파악 및 요약

학교매점(가칭) — 협력사 폐쇄형 기부몰 구축

시설이 돈을 받아 직접 사던 구조를, 기부자가 물품을 직접 결제해 시설이 물건을 받는 구조로 바꾼다. 결제는 학교 플랫폼 법인이 아닌 별도 협력사(학교매점 운영 사업자)가 받고, 그 구매영수증이 시설의 기부영수증 발급 근거가 된다.

1 무엇이 달라지는가

지금 (1차 MVP)

  • 기부자가 금액을 후원 → 시설 전용계좌에 귀속
  • 목표 달성 후 운영자가 한 번에 통합구매
  • 기부자는 알림장으로 사후 확인
  • 시설이 구매 내역을 직접 대조해 기부영수증 발급

바뀔 모습 (2차 MVP)

  • 기부자가 물품 자체를 학교매점에서 결제
  • 결제 즉시 협력사가 소싱사이트에 발주
  • 기부자는 결제 시점에 무엇이 전달되는지 확인
  • 시설은 전달받은 구매영수증을 확인해 발급만

바꾸는 이유 세 가지 — ① 기부자가 결제 시점에 사용처를 즉시 확인 ② 재고를 아무도 떠안지 않음(B2B 위탁 소싱) ③ 시설의 기부영수증 발급 업무가 "만드는 일"에서 "확인하는 일"로 단순화.

2 핵심은 세 갈래 길을 분리하는 것

이 기획의 설계 원리는 물건·돈·영수증이 서로 다른 경로로 흐른다는 것이다. 이 분리를 이해하지 못하면 도메인 설계가 어긋난다.

흐름 경로 학교 플랫폼 법인의 역할
물건 B2B 소싱사이트 → (물류) → 협력교실(시설) 수령확인 기록만
돈 기부자 → 협력사(학교매점) 계좌 → 소싱사이트 발주대금 돈을 일절 보관하지 않음
영수증 학교매점이 구매영수증 발행 → 시설 → 시설이 기부영수증 발급 → 기부자 전달·상태 추적만

전체 파이프라인

[협력교실] 카탈로그에서 장바구니 담아 제출
      ↓
[운영팀]  품목·수량·가격 검수  →  "프로젝트"로 발행  →  공개 예약
      ↓
[기부자]  프로젝트 상세에서 물품 선택 → 학교매점 장바구니 → 결제
      ↓
[협력사]  결제 수령(별도 사업자·별도 계좌) + 구매영수증 발행
      ↓
[소싱사이트] 발주 → 물류·배송
      ↓
[협력교실] 수령확인(검수 사진 업로드)
      ↓
[협력사 → 시설] 구매영수증 전송
      ↓
[시설]    구매영수증 확인 → 기부영수증 발급
      ↓
[플랫폼]  알림장 발행 · 성적표/학년 점수 반영 · 학급일지 기록

3 영수증 4단계 — 기부자에게 보여줘야 하는 상태

기획이 가장 강조하는 UX 포인트. "구매하면 자동으로 기부영수증이 나온다"고 오해시키면 안 된다. 시설의 확인 절차가 남아 있음을 단계 표시로 드러내야 한다.

발행됨→ 시설 전송됨→ 시설 확인 중→ 기부영수증 발급됨
주의 — 이 4단계는 기부자 마이페이지에서 직접 추적 가능해야 한다. 기획서 8장의 리스크 완화 항목이자, 5-3 화면 흐름에 명시된 요구사항이다.

4 화면 흐름 요약 (3주체)

① 협력교실(시설) — 물품 제출

핵심 요구: "장바구니 담듯 쉽게". 긴 서술형 입력 금지, 짧은 태그·메모로 간소화.

  1. 카탈로그 탐색 (카테고리 → 성별·연령·사이즈 필터)
  2. 장바구니 담기 (수량 + 필요 이유 짧은 메모)
  3. 제출 전 요약 화면 (리스트 + 예상 총액, 수정 가능)
  4. 제출 → "검수 대기 중" 전환 + 담당자 알림

② 운영팀(교무실) — 검수·발행

  1. 검수 대기 목록 (시설별, 제출일순)
  2. 상세 검수 (적정성·중복·금지품목)
  3. 승인 / 보류 / 반려(사유 자동 전달)
  4. 승인분을 하나의 프로젝트로 묶어 발행
  5. 공개 예약 (노출 시작일·마감일)

③ 기부자 — 참여부터 결제·추적까지

  1. 홈 "이번 프로젝트" 배너 → 프로젝트 상세(진행률·남은 물품 카드·참여자 수)
  2. 물품 카드 개별 선택 → 학교매점 장바구니(여기서 협력사 화면으로 전환)
  3. 학교매점 결제 시스템으로 결제 (플랫폼 계좌와 분리)
  4. 결제 완료 = 참여 확정 + 구매영수증 발행 → 즉시 플랫폼 화면 복귀
  5. 성적표·학년 점수 반영, 공유 유도, 마이페이지에서 영수증 4단계 추적
이탈 방지 4원칙 (기획 5-3) — ⓐ 학교매점 화면도 디자인 톤 통일 ⓑ 결제 상단에 "학교 플랫폼과 함께하는 협력 매점입니다" 안내 ⓒ 결제 후 지체 없이 복귀 ⓓ 영수증 진행 상태 추적 제공

5 현재 코드베이스와의 격차

donation-platform-server의 실제 도메인을 기준으로 대조했다. 결론부터 말하면 1차 MVP 도메인 대부분이 재정의 대상이며, 특히 PurchaseOrder는 의미가 뒤집힌다.

현재 도메인 지금 하는 일 2차 MVP에서 판정
Facility 시설 마스터 CRUD (관리자 전용) 시설 담당자 계정이 직접 로그인해 제출하는 주체가 됨 → 계정↔시설 소유권 매핑 및 FACILITY_MANAGER 역할 필요 확장
DonationItem 시설 직속 자유입력 물품
name·category·goalAmount·raisedAmount·targetQuantity 		카탈로그 SKU 참조로 전환. 이름·가격을 시설이 쓰는 게 아니라 상품에서 끌어옴. 사이즈·색상 옵션 개념 추가. 상위에 Project가 생김 재정의
Contribution status = RECORDED 단일값.
주석 그대로 "실제 결제 없이 참여를 기록만" 		협력사 실결제와 연결. PAID·REFUNDED 확장은 코드 주석에서도 "전문가 검토 관문 통과 후"로 이미 유보돼 있음 관문
PurchaseOrder 목표 달성 후 운영자가 외부에서 한 번에 사고 결과를 기록. donation_item_id UNIQUE로 중복구매 차단 의미 반전 — 기부자 결제 시점마다 소싱사이트로 발주. "물품당 1건" UNIQUE 제약이 성립하지 않음 반전
ItemStatus RECRUITING→LOCKED→BUYING→SHIPPING→RECEIVED→RECEIPTED 제출·검수 단계(DRAFT·SUBMITTED·REVIEWING·REJECTED)가 앞에 붙고, 영수증 단계는 4단계로 쪼개짐 확장
PurchaseOrder.evidenceNo 구매증빙 번호. 주석에 "세액공제 영수증 아님" 명시 협력사가 발행하는 정식 구매영수증으로 승격 + 시설 전송·확인 상태 추적 신규
— 없음 — 카탈로그(상품·옵션·재고), 장바구니, 제출건/검수, 프로젝트, 주문/결제, 구매영수증, 소싱사이트 연동, 원가/판매가(마진) 신규
menu·role·permission 메뉴·권한 체계 (관리앱 메뉴 관리까지 완료) 그대로 활용. 학교매점 메뉴·시설담당자 권한만 추가 유지
notification 알림 기반 제출·반려·검수완료·수령확인 알림에 재사용 유지

6 해야 할 일 — 관문과 무관하게 지금 시작할 수 있는 것

먼저 짚을 것 — 기획 10장은 "① 협력사 설립 확정 → ② 전문가 검토 → ③ 소싱사이트 선정 → ④ 개발" 순서로 적혀 있다. 하지만 실제로는 ④의 앞부분(제출·검수·프로젝트 발행)은 돈과 영수증을 건드리지 않으므로 관문과 무관하게 병행 착수할 수 있다. 반대로 결제·영수증은 관문 통과 전 착수하면 안 된다 — 구조가 확정되기 전에 만들면 전부 버리게 된다.

Phase A — 지금 바로 착수 가능 관문 무관

# 할 일 내용
A-1 카탈로그 도메인 설계 상품(SKU)·옵션(사이즈/색상)·재고·카테고리(의류/잡화). 원가와 판매가를 필드로 분리하되 원가·마진은 외부 응답 DTO에 절대 노출하지 않는 것을 설계 단계에서 못박음. 소싱사이트 미정이므로 어댑터 인터페이스 + 수동입력 구현체로 먼저 만든다
A-2 시설 담당자 계정·권한 FACILITY_MANAGER 역할 추가, 계정↔시설 소유권 매핑, 자기 시설 데이터만 접근하는 인가 규칙. 기존 role·permission 체계 재사용
A-3 장바구니 → 제출 흐름 Cart / ItemRequest(제출건) 도메인. 제출 시 스냅샷 저장(가격 변동 대비). 시설용 제출 화면 3단계(탐색 → 담기 → 요약)
A-4 운영팀 검수 화면 검수 대기 목록 · 상세 검수 · 승인/보류/반려(사유 자동 전달). 금지 품목 룰(사치품·상품권· 위험물)을 카탈로그 등록 단계에서 차단하는 게 1차 방어선
A-5 Project 도메인 신설 승인 품목 묶음 + 프로젝트명 + 공개 시작일/마감일 + 진행률 집계. 현재 DonationItem이 Facility 직속인 구조를 Facility → Project → Item으로 한 단계 넣는 마이그레이션 포함
A-6 기부자 프로젝트 상세 화면 진행률 게이지·남은 물품 카드·참여자 수. 장바구니 담기까지만 만들고 결제 직전에서 멈춘다 (Phase C와 접합)

Phase B — 관문(선행 결정) 개발 아님

# 확인 항목 막히는 개발
B-1 협력사 설립 방식 (신규 법인 / 기존 커머스 제휴) 결제 연동 전체, 정산 설계
B-2 6-1 구조 성격 확정 — 구매위임형인지 다른 거래인지 주문·결제 도메인의 당사자 모델링
B-3 세무 검토 — 구매영수증 → 기부영수증, 현물기부 가액 산정 영수증 도메인 전체, 4단계 상태
B-4 B2B 소싱사이트 선정 (재고 책임·품절·반품 주체) A-1 어댑터의 실제 구현체, 실시간 재고 연동
B-5 통신판매업 신고 등 협력사 커머스 요건 결제 오픈 시점
B-6 마진율·재기부/운영비 배분 (비공개 원칙 유지) 회계 처리, 정산 리포트

Phase C — 관문 통과 후 착수 금지

# 할 일 내용
C-1 주문·결제 도메인 협력사 PG 연동. Contribution.status에 PAID·REFUNDED 추가 (코드 주석이 이미 "관문 통과 후"로 유보해 둔 지점)
C-2 구매영수증 도메인 + 4단계 추적 발행 → 시설 전송 → 시설 확인 → 기부영수증 발급. 기부자 마이페이지 추적 UI 포함. 자동 발급으로 읽히는 문구는 전면 금지
C-3 인웹 전환 UX 학교매점 화면 디자인 톤 통일, 결제 상단 협력사 안내 문구, 결제 후 즉시 복귀
C-4 소싱사이트 실발주 연동 A-1 어댑터에 실제 구현체 주입, 실시간 재고로 품절 옵션 자동 차단
C-5 PurchaseOrder 재설계 "물품당 1건 통합구매"(UNIQUE) → "결제 건별 발주"로 전환하는 마이그레이션
C-6 완결 루프 재연결 수령확인(검수 사진) → 알림장 → 성적표·학년 점수 → 학급일지(마진 비공개)

Phase D — 시범 운영

소규모 시설 1곳 · 프로젝트 1건으로 제출부터 기부영수증 발급까지 전 과정 점검 후 확대 여부 결정. (기획 10장 5~6)

7 설계 시 놓치면 안 되는 제약

제약 구현상 의미
플랫폼 법인은 돈을 보관하지 않는다 플랫폼 서버는 결제 금액을 정산·보관하는 주체가 아니다. 결제 결과를 참조·기록만 한다. 잔액·출금 개념을 플랫폼 도메인에 만들면 안 됨
마진율 비공개 원가·마진 필드는 존재하되 공개 API 응답 DTO에서 원천 배제. 학급일지·완결 리포트 에도 마진 수치 미노출
기부영수증은 자동이 아니다 시스템은 시설의 발급 행위를 대행하지 않는다. 상태는 시설의 액션으로만 전진
시설별 발급 자격이 다름 Facility에 기부영수증 발급 자격 플래그 필요. 자격 없는 시설은 4단계 UI를 다르게 안내
재고를 떠안지 않는다 플랫폼/협력사 DB에 재고 수량을 소유하지 않고 소싱사이트 조회 결과를 캐시로만 취급. 품절은 프로젝트 등록 시점에 차단

8 한 줄 결론

A-1 ~ A-6(카탈로그·시설담당자·제출·검수·프로젝트·상세화면)은 지금 착수한다. 결제와 영수증(C)은 협력사 구조와 세무 검토(B-1~B-3)가 확정되기 전까지 손대지 않는다. 현재 코드의 Contribution.status = RECORDED와 "세액공제 영수증 아님" 주석이 바로 그 관문의 위치를 이미 표시해 두고 있다.
