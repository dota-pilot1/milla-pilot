통합 구매 — 테이블 설명

ERD(테이블 설계.mmd)의 각 테이블을 컬럼 단위로 설명. 신규는 PURCHASE_ORDER 하나, 나머지(FACILITY·DONATION_ITEM·CONTRIBUTION)는 관계 맥락용 요약이다.

0. 관계 한눈에

FACILITY1 ─< N DONATION_ITEM1 ─< N CONTRIBUTION
DONATION_ITEM1 ── 0..1 PURCHASE_ORDER (신규) 물품당 통합구매 최대 1건

1. purchase_orders 신규 · Step5

목표달성(LOCKED) 물품을 한 번 통합구매한 구매 기록. 판매처·실구매액·증빙을 담는다.

컬럼타입제약설명
idBIGINTPK IDENTITY기본키
donation_item_idBIGINTFK UNIQUE NOT NULL대상 물품. UNIQUE = 물품당 1건 → 중복구매 원천 차단(§15 "중복구매 0건")
vendorVARCHAR(200)NOT NULL판매처(구매한 곳)
purchased_amountBIGINTNOT NULL실제 구매액. 목표액과 다를 수 있음(가격변동은 차기 D-002)
evidence_noVARCHAR(200)구매증빙(주문·송장 번호 등). 세액공제 기부금영수증 아님(F-015)
noteVARCHAR(500)메모(예외·비고)
ordered_atTIMESTAMP구매(기록) 일시. 미입력 시 생성시각
created_atTIMESTAMPNOT NULL@CreationTimestamp
updated_atTIMESTAMPNOT NULL@UpdateTimestamp
핵심 제약: donation_item_id UNIQUE 하나로 "중복구매 0건"을 DB 레벨에서 보장. 동시 실행(레이스)도 유니크 위반으로 막힌다(step3 락과 같은 철학).
실결제 없음(MVP): 이 레코드는 운영자가 "구매했다"고 기록하는 것. 실제 결제·정산은 전문가 검토 관문 뒤(§9 C-003).

단계와의 관계 (이중관리 금지)

통합구매 실행 → PURCHASE_ORDER 생성 + donation_item.status = BUYING. 이후 배송·수령·영수증 단계는 donation_item.status(6단계)가 단일 출처. PURCHASE_ORDER는 구매 상세만 담고 단계 값을 중복 보관하지 않는다.

2. donation_items 기존

시설의 준비물 목표(품목). 통합구매의 대상이자 이행 단계(status)의 주체.

컬럼설명(통합구매 관점)
goal_amount / raised_amount목표액 / 모인액(Σ Contribution). raised ≥ goal → LOCKED에서 통합구매 가능
status이행 단계 단일 출처 — RECRUITING → LOCKED → BUYING(통합구매 실행 시) → SHIPPING → RECEIVED → RECEIPTED
관계1 : 0..1 PURCHASE_ORDER — 통합구매 안 했으면 0, 했으면 1건
가드: status = LOCKED일 때만 통합구매 실행 허용. RECRUITING(미달성)·이미 PO 있는 물품은 거부.

3. contributions 기존

후원자별 참여 기록(기부 원장). 통합구매와 직접 FK는 없지만, 이 합이 목표 도달=통합구매 조건을 만든다.

컬럼설명
donation_item_id · donor_id어느 물품에 · 누가
amount참여 금액. Σ amount = donation_item.raised_amount
statusRECORDED(MVP: 결제 없이 기록). 차기 PAID/REFUNDED

4. facilities 기존

후원 대상 시설. 물품의 상위. 통합구매는 물품 단위라 시설과 직접 관계는 없다(물품 경유).

컬럼설명
name · region · type시설명 · 지역 · 유형
status · verified표시 상태 · 자격확인

5. 관계·제약 정리

  • DONATION_ITEM 1 : 0..1 PURCHASE_ORDER — donation_item_id UNIQUE.
  • 중복구매 0(§15) = UNIQUE + 서비스 가드(LOCKED만).
  • 이행 단계는 donation_item.status 단일 출처. PURCHASE_ORDER는 구매 상세(판매처·금액·증빙).
  • 삭제 정책: 시설 삭제 → 물품 삭제 시 소속 PURCHASE_ORDER도 함께(ON DELETE CASCADE).
  • F-015: evidence_no는 구매·배송 확인 자료일 뿐 세액공제 영수증 아님(문구 상시).
