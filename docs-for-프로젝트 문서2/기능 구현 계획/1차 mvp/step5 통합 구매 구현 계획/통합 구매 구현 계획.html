Step 5 통합 구매 관리자(Tauri)

통합 구매 구현 계획

목표달성(잠금)된 물품을 한 번만 통합구매하고, 판매처·실구매액·증빙을 기록해 배송·수령·영수증으로 잇는다. 도식: 통합 구매 프로세스 도식화.mmd.

1. 기획 근거 (있음 — 핵심)

  • §5 한 문장: "다 모이면 플랫폼이 추가 결제를 막고, 시설(또는 적법 위임 구매자)이 한 번만 사서 시설로 보낸다"
  • §8 이행 흐름: 목표달성 잠금 → 위임받은 구매자가 한 번만 통합구매 → 판매처에서 시설로 직배송
  • §9 조건부 C-002(통합구매 주체·구매위임 R-003), C-003(실제 결제 없이 가상 리허설)
  • F-013 공동충당·목표잠금·한 번의 통합구매 · F-015 구매증빙 ≠ 세액공제 영수증
  • §15 지표: "목표잠금 후 중복구매 0건", 영수증 발급자료 완전성 100%
즉 통합구매는 "물품당 딱 한 번, 운영자가 실행·기록"하는 절차. 실결제는 MVP 범위 밖(관문 뒤) — 구매를 "기록"한다(가상 리허설).

2. 무엇을 구현하나 — PurchaseOrder (통합구매 기록)

물품당 1건의 구매 기록. DB UNIQUE(donation_item_id)로 중복구매를 원천 차단(§15 0건).

컬럼설명
idPK
donation_item_idFK · UNIQUE (물품당 1건 = 중복구매 방지)
vendor판매처
purchased_amount실제 구매액(목표액과 다를 수 있음 — 가격변동 D-002는 차기)
evidence_no구매증빙(주문번호·영수증 등). 세액공제 영수증 아님(F-015). 송장·배송추적은 step6로 분리
note메모
ordered_at / created_at구매·기록 일시
이행 단계(배송·수령·영수증)는 이미 있는 item.status(6단계)로 추적한다. PurchaseOrder는 구매 상세(판매처·금액·증빙)를 담고, 단계는 item.status가 단일 출처. 이중 관리 안 함.

3. 상태 관계

모집중(RECRUITING)
  └ 후원 공동충당 → 목표 도달 → 자동 잠금
목표달성(LOCKED)                 ← 통합구매 실행 가능
  └ [통합구매 실행: 판매처·금액·증빙]
     → PurchaseOrder 생성(1건) + item.status = BUYING
배송중(SHIPPING) → 수령확인(RECEIVED) → 영수증(RECEIPTED)
     (단계 전환은 step4 advancer / 상태 관리로)

가드: LOCKED 상태에서만 통합구매 실행 가능. RECRUITING(미달성)·이미 PO 있는 물품은 거부.

5. Tauri 화면 — "통합 구매" 메뉴

ADMIN_PURCHASE_ORDERS(/purchase-orders) 화면. 운영자의 작업 큐.

  • 목표달성(LOCKED) 물품 목록 = "구매 대기" 큐 (시설·물품·목표액·모인액)
  • 행 [통합구매 실행] → Drawer: 판매처·실구매액·증빙 입력 → 확인(ConfirmDialog) → PO 생성 · item BUYING
  • 이미 구매된 물품은 구매 완료 표시(판매처·금액·증빙) — 재실행 불가
  • primitive 재사용: Panel · DataTable · Drawer · FormField · Input · ConfirmDialog · StatusBadge · EmptyState
F-015 안내 상시: 화면·증빙에 "구매증빙(주문·송장)은 배송 확인 자료일 뿐, 세액공제 기부금영수증이 아닙니다" 문구.

6. 정합성 · 제약

  • 중복구매 0건(§15) — donation_item_id UNIQUE + 서비스 가드. 동시 실행도 DB 유니크로 방지.
  • 실결제 없음(MVP) — 구매를 "기록"만. 실제 결제·정산은 전문가 검토 관문 뒤(§9 C-003).
  • 구매증빙 ≠ 영수증(F-015) — 문구 상시.
  • 가격변동·환불(D-002) — 차기. 이번엔 실구매액 기록까지만.
