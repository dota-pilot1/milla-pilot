Step 6 배송 추적 별도 구현 · 자동화 우선

배송 추적 구현 계획

통합구매(step5) 이후 판매처 → 시설 직배송을 추적한다. 원칙: 관리자 수동 상태변경 금지 — 사람이 넣는 건 송장번호 1회뿐, 배송 단계는 딥링크/집계 API로 자동.

1. 원칙 · 기획 근거

  • §8 이행 흐름: 판매처에서 시설로 직배송 → 시설 수령확인
  • §7 1차 파일럿: 배송·수령 관리(tracking) 포함
  • 실제 택배사 연동은 원래 "파일럿 이후"(파트너 R-002·R-003)지만, 무료 집계 API/딥링크로 지금도 저비용 구현 가능
핵심 원칙: 관리자가 배송중→완료를 손으로 클릭하는 방식 금지(비효율). 송장번호만 1회 입력하면 상태는 자동으로 따라온다.

2. 데이터

step5 PURCHASE_ORDER에 배송 필드 추가(별도 테이블도 대안). 구매증빙(주문번호)과 송장(운송장번호)은 구분.

컬럼설명
courier택배사 코드(CJ대한통운·우체국·한진 등) — 관리자 1회 입력
tracking_no운송장번호 — 관리자 1회 입력
delivery_status배송 단계 캐시(READY·IN_TRANSIT·OUT_FOR_DELIVERY·DELIVERED) — API 동기화 값
last_synced_at마지막 배송상태 조회 시각
배송 이벤트 이력(집화→간선→…)까지 저장하려면 Shipment + ShipmentEvent 테이블로 분리 가능. MVP는 위 캐시 필드로 충분.

3. 2단계 구현

단계방식비용자동화
Phase 1 · MVP택배사+송장번호 저장 → 조회 페이지 딥링크. 웹 "송장" 클릭 → 택배사/통합조회 페이지 열림(드로워 or 새 탭)0사용자 클릭 조회
Phase 2택배 집계 API 연동 → 서버가 배송상태 pull → 드로워에 단계 표시. 폴링/온디맨드 동기화무료~소액완전 자동

API 후보 (집계형 — 개별 택배사 X)

  • delivery-tracker (오픈소스 · tracker.delivery · GraphQL) — 무료, 파일럿에 얹기 쉬움 → MVP 추천
  • 스윗트래커 / 스마트택배 API — 국내 표준, 안정적, 무료 티어 → 프로덕션 선호
개별 택배사 API를 여러 개 붙이지 말 것. 한국은 택배사가 많아 집계 API 하나로 전 택배사 커버한다.

4. 수령확인은 사람 (신뢰 장치)

자동 추적으로 배송완료(DELIVERED)가 떠도, 시설 수령확인(RECEIVED)은 시설/관리자가 확인하는 별도 스텝(§8). 배송완료 ≠ 실물 수령확인.

  • 배송중 상태 = 자동 (딥링크/API)
  • 수령확인 = 사람 1클릭 → item.status = RECEIVED (기존 단계 관리 재사용)

5. 화면
