Step 4 후원 내역 · 기부 원장 관리자(Tauri)

후원 내역(기부 원장) 구현 계획

관리자가 물품·시설별로 "누가 얼마 후원했는지"를 조회하는 기능. 후원자 화면(내 후원 내역)의 운영자 버전이며, 기획이 운영자에게 요구하는 "후원자별 기록으로 추적"을 채운다.

1. 기획 근거

  • §3 운영자 문제→변화: "하나의 후원자별 기록(기부 원장)과 공통 상태값으로 추적"
  • F-014: 후원자별 결제 식별·기부 원장 관리
  • §9: 플랫폼은 식별·기록·대조·잠금·상태관리·검수를 맡는다
기록은 이미 Contribution 엔티티로 쌓이고 있다(Step2). 이번 스텝은 그걸 관리자가 읽는 뷰 + 조회 API만 추가한다. 새 엔티티 없음.

2. 백엔드

대상내용구분
ContributionRepositoryfindByDonationItem_IdOrderByCreatedAtDesc(itemId) — 후원자 fetch join추가

  { "id": 12, "amount": 300000, "createdAt": "...", "donor": { "id": 5, "username": "후원자01", "email": "donor01@test.com" } },
  { "id": 13, "amount": 120000, "createdAt": "...", "donor": { "id": 6, "username": "후원자02", "email": "donor02@test.com" } }

개인정보: 후원자(성인)의 username·email은 관리자만 본다. 아동 정보와 무관(§4 아동은 후원자 아님). 공개 웹에는 절대 노출하지 않는다.

3. 프론트 (Tauri)

메인 — 물품별 기부 원장 (추천)

시설관리 물품 패널의 각 물품 행에 "후원 내역" 버튼 → Drawer로 그 물품의 후원자 목록 표시. 컨텍스트(어느 물품)와 함께 봐야 의미 있고, Drawer·DataTable primitive를 재사용한다.

요소내용구분
entities/contribution (Tauri)타입 + fetchItemContributions(itemId)추가
물품 행 "후원 내역" 버튼 + Drawer후원자·금액·일시 DataTable + 합계primitive 재사용
빈 상태EmptyState "아직 후원이 없습니다"재사용

옵션 — 독립 "후원 내역" 메뉴 화면 (차기)

사이드바 후원 내역(/contributions) 메뉴를 시설·물품 필터가 있는 전용 조회 화면으로. MVP는 위 물품별 Drawer로 충분하므로 후순위.

4. 화면 흐름

시설 관리 → 시설 선택 → 물품 패널
  → 물품 행 [후원 내역] 클릭
  → Drawer: "초등 학습 문제집 세트 · 후원 내역"
       후원자01   300,000원   2026-07-17
       후원자02   120,000원   2026-07-17
       ─────────────────────────────
       합계        420,000원  (2명)

5. authz 연계 (같이/직후)

후원 내역 조회는 관리자만 봐야 한다. 현재 후원 관리 API는 "로그인만 하면" 접근 가능(갭). 이 스텝과 함께 또는 직후에:

  • 후원 관리 엔드포인트(/api/facilities 변경, /api/donation-items 변경, /contributions 조회)를 ROLE_PLATFORM_ADMIN 전용으로 제한
  • SecurityConfig requestMatchers(...).hasRole("PLATFORM_ADMIN") 또는 메서드 보안
공개 GET(/api/facilities/** 목록·물품)은 그대로 열어두고, 후원 내역·CRUD·상태변경만 관리자로 막는다.
