# 후원 모델 정의서

> 시설 물품 **공동 지정기부** 플랫폼의 서비스 모델·자금 흐름·정책을 확정한 문서.
> 기획·개발 공통 기준선이며, 여기 없는 기능은 "안 만드는 것"으로 간주한다.

---

## 1. 한 줄 정의

**후원자가 특정 물품을 지정해 공동으로 후원하면, 목표 달성 시 단체가 그 물품을 구입해 시설로 전달하고 증빙을 공개하는 서비스.**

- 후원자는 **물건을 사는 것이 아니라 용도를 지정해 후원**한다.
- 물품의 소유·구매 주체는 **단체**다. 후원자에게 소유권·지시권이 없다.

---

## 2. 주간 라운드 (게임형 모금)

물품이 몇 개든 **운영 부담이 상수로 고정**되는 것이 이 모델의 핵심.

```
월  이번 주 물품 3건 공개 (라운드 시작)
    ↓
    간편결제로 공동 모금        ← 이 구간 취소·환불 자유
    ↓
일  23:59 마감
    ├─ 달성 → 🔔 "모금 완료" 알림 → LOCKED (환불 잠금)
    │         ↓
    월        관리자 구매 → 시설 직배송
    │         ↓
    →        🔔 "전달 완료" 알림 + 영수증·송장 공개
    │
    └─ 미달 → 🔔 "이번엔 아쉽게" → 포인트 환급
```

**규칙**
- 주당 물품 **3건** 고정 (비교적 고가 물품)
- **전액 달성 조건(all-or-nothing)** — 미달 시 구매하지 않음
- 마감 = **확정선**. 이후 환불 불가
- 초기 몇 라운드는 **확실히 달성 가능한 금액**으로 성공 경험을 쌓는다
- 시설이 여러 곳이면 **노출 순번 규칙**을 별도로 정한다

## 3. 결제와 자금 흐름

### 결제 수단
- **주력: 간편결제**(카카오페이·네이버페이·토스페이·카드) — 이탈률 낮고 환불이 API 한 번
- **보조: 가상계좌** — 수수료 저렴, 기부 정서에 맞음. 환불 시 계좌 송금 필요
- PG는 **포트원** 또는 **토스페이먼츠 직연동**. 계약 전이라도 **테스트 키로 전 기능 구현 가능**

### "공동 모금"의 실체
PG에 공동 모금 기능이 있는 게 아니다. **개별 결제 N건에 물품 ID를 붙여 저장하고 합산**하는 우리 집계 로직이다.

```sql
SELECT SUM(amount) FROM contributions WHERE donation_item_id = ?;
```

### 자금 흐름
```
후원자 결제 → PG 수납·보관 → (정산 D+N, 수수료 차감) → 단체 명의 계좌 → 물품 구매
```

- PG 계약은 **반드시 단체/법인 명의**. 개인 명의는 소득·투명성 문제 발생
- **정산 시차**로 모금 완료 직후엔 현금이 없다 → **법인카드 선지출 후 정산금으로 상환**
- → **한 라운드분 운영자금(cash float)이 필요**

### 수수료 — 목표액 산정 주의
모금액 ≠ 입금액. **PG 수수료(정률)가 차감**되므로 목표액을 물건값과 동일하게 잡으면 매번 모자란다.

> **목표액 = 예상 구매가 + 수수료 + 여유 5~10%**

가격 변동까지 함께 흡수된다. PG 상담 시 **비영리·기부 수수료 우대** 여부를 반드시 확인할 것.

## 4. 환불 정책

| 사유 | 처리 |
|---|---|
| 모금 미달 | **포인트 환급(기본)** + 계좌 환불 신청 가능 |
| 착오·중복 입금 | **계좌 환불(무조건)**, 차액은 운영 흡수 |
| 마감 **전** 취소 | 자유. 포인트/계좌 중 후원자 선택 |
| 마감 **후** | **불가** (LOCKED) — 후원 화면에 사전 고지 필수 |

### 포인트 규칙 (중요)
- **충전 기능을 만들지 않는다.** 선불 충전은 전자금융거래법상 선불전자지급수단 이슈(미상환 잔액 보호)를 유발한다.
- 포인트는 **모금 실패 환급 전용**.
- **언제든 계좌로 출금 가능해야 한다.** 이 원칙을 지키면 "돈을 묶어두는 구조"가 아니게 되어 규제·분쟁 리스크가 사라진다.

## 6. 만들지 않는 것 (금지 목록)

방어선이다. "후원자 편의"를 명분으로 슬쩍 들어오면 규제가 통째로 딸려온다.

### 서비스 범위
| 항목 | 이유 |
|---|---|
| ❌ 자체 물류센터 / 선구매 재고 | 모금 실패 = 손실. 창고+인건비 연 5천만원 = 물품 160개 |
| ❌ 풀필먼트·3PL 연동 | 재고가 없으므로 해당 없음. 쇼핑몰이 시설로 직배송하면 끝 |
| ❌ 현물(중고품) 수거 | 집하·검수·재분배가 필요한 별도 사업 |
| ❌ 외부 쇼핑몰 결제 연동 | 소비자 대신 주문하는 API는 국내에 존재하지 않음 |
| ❌ 쿠팡 파트너스 API (초기) | 상품 검색만 가능. 승인 심사 + 커미션 이해상충 대비 실익 없음 |
| ❌ 오픈뱅킹·계좌 스크래핑 조회 | 입금자 매칭이 안 되므로 실익 없음 |
| ❌ 포인트 충전 | 선불전자지급수단 규제 |

### 구매대행이 되어버리는 조건 — 하나라도 넣으면 안 됨
- 후원자가 **배송지를 지정**할 수 있다
- 후원자가 **물건을 직접 받을** 수 있다
- 실구매 **차액을 후원자에게 환급**한다
- 마감 후 후원자가 **물품을 변경·지시**할 수 있다
- **대행 수수료**를 별도로 받는다
- 후원자 화면에 **구매 버튼**이 노출된다 (관리자 전용이어야 함)

"쇼핑몰"로 프레이밍하면 통신판매중개업 신고·전자상거래법 표시의무·청약철회 14일이 따라붙는다.

| ❌ 쓰지 않는다 | ✅ 쓴다 |
|---|---|
| 구매하기 / 장바구니 | 후원하기 |
| 주문내역 | 내 후원 내역 |
| 결제금액 | 후원금액 |
| 배송조회 | 전달 현황 |
| 구매대행 / 기부대행 | 공동 지정기부 |
| "자동으로 구매됩니다" | "모금 완료 즉시 구매를 진행합니다" |

### 필수 고지 문구
> 후원금은 지정하신 물품 구입에 사용되며, 잔액 발생 시 해당 시설의 다른 필요 물품에 사용됩니다. 결제 수수료를 제외한 전액이 물품 구입에 사용됩니다.

## 8. 자동화 범위

| 단계 | 자동 | 비고 |
|---|---|---|
| 결제 | ✅ | PG 연동 |
| 달성 판정 → LOCKED | ✅ | 일요일 마감 배치 |
| 관리자 구매 지시 알림 | ✅ | |
| **실제 발주** | ❌ | **사람이 클릭** — 주 1회, 3건, 5분 |
| 구매 결과·송장 입력 | ❌ | 관리자 모달 2회 |
| 배송 추적 | ✅ | 딜리버리트래커 웹훅 |
| 후원자 알림 | ✅ | |

후원자가 체감하는 흐름은 **100% 자동**이다. 완전 자동 발주는 도매 드랍쉬핑 API(오너클랜·도매매 등)로만 가능하지만, 상품 제한·재고·반품 부담이 커서 물량이 늘기 전에는 도입하지 않는다.

### 관리자 화면 (Tauri)
```
┌─────────────────────────────────┐
│ 🧺 세탁기          ✅ 모금완료   │
│ 목표 800,000 / 모금 800,000     │
│ [상품 페이지 열기] [구매 완료 입력] │
└─────────────────────────────────┘
```
- 상품 페이지 열기 → 저장된 `vendorUrl`
- 구매 완료 입력 → 실구매액·주문번호·영수증 → `BUYING` 전이 + 후원자 알림
- 이후 운송장 등록 → 배송추적 → 전달 완료 알림

## 9. 물품 유형과 잔액 관리

### 9-1. 물품 유형

| | **단품형** | **수량형** |
|---|---|---|
| 예 | 세탁기 1대 | 연탄, 기저귀, 노트북 N대 |
| 목표 | 물건값 | `unitPrice` × `targetQuantity` |
| 잔액 | 소액 | **단가만큼 발생 가능** |

`targetQuantity`는 기존 필드. **`unitPrice` 추가 필요.**
잔액 크기는 단가에 비례한다 — 연탄(4,500원)은 무시 가능, 노트북(100만원)은 심각.

### 9-2. 수량형은 최소/희망 2단 목표

수량형에 전액 달성 조건을 그대로 적용하면 실패율이 지나치게 높다.

```
노트북 (단가 100만원)
  최소 목표  3대 =   3,000,000   ← 이것만 넘으면 구매 진행
  희망 목표 10대 =  10,000,000

결과 7,800,000 모금 → 7대 구매 → 잔액 800,000
```

- 최소 목표 **미달** → 전액 환급 (4절 규칙)
- 최소 목표 **달성** → 살 수 있는 만큼 구매, 나머지는 잔액 풀로

단품형은 기존대로 전액 달성 조건.

### 9-3. 가격 오차 — 맞추려 하지 않는다

쿠팡 등 온라인 가격은 계속 변한다. **정확히 맞추는 것을 목표로 하지 않고, 안 맞아도 문제가 없게** 설계한다.

```
목표액 = 조사 시점 가격 + PG 수수료 + 여유 10%
```

| 마감 후 실제 가격 | 처리 |
|---|---|
| 내렸다 | 잔액 → 시설 풀 |
| 소폭 올랐다 | 여유분이 흡수 |
| 많이 올랐다 (수량형) | 1개 덜 사고 잔액 이월 |
| 많이 올랐다 (단품형) | 대체 상품 · 잔액 풀 충당 · 다음 라운드 이월 중 관리자 판단 |

후원자에게 **정확한 가격을 약속하지 않는다** — "약 800,000원(예상)"으로 표기하고, 결과에서 실제 숫자를 공개한다. 오차와 영수증이 함께 보이는 쪽이 신뢰를 만든다.

### 9-4. 시설별 잔액 풀

**잔액은 절대 후원자에게 환급하지 않는다.** 환급하는 순간 구매대행이 된다(6절).
대신 **시설별 잔액 풀**에 적립하고 전액 공개한다.

```
🏫 OO보육원 이월 잔액 350,000원 · 집행 기한 2026-10-19
   +800,000  노트북 라운드 잔액        (7/19)
   −450,000  겨울 이불 6채 구입 📎영수증 (7/26)
```

**집행 규칙**

| 잔액 | 처리 |
|---|---|
| 소액 (~1만원) | 그대로 이월. 억지로 소진하지 않는다 |
| ~20만원 | 다음 라운드 물품에 우선 투입 |
| **20만원 초과** | 초과분을 **상시 소모품**으로 즉시 집행해 20만원 이하로 낮춘다 |
| **3개월 경과** | 강제 집행. 미집행 시 시설 일반후원 전환 + 공지 |

**상시 소모품** — 시설이 미리 등록해둔 상시 필요 품목(화장지·세제·간식·학용품 등)에서만 집행한다.
잔액을 털기 위해 아무거나 사는 것이 아니라, **시설이 필요하다고 등록한 것**이어야 한다.

**집행 범위 (엄수)**

| | 가능 | |
|---|---|---|
| 같은 시설의 다른 물품 | ✅ | 고지된 지정 범위 안 |
| 다른 시설로 이전 | ❌ | 용도 이탈 |
| 운영비·인건비 전용 | ❌ | 가장 위험 |
| 현금으로 시설 전달 | ⚠️ | 물품 지정 취지 이탈. 권장하지 않음 |
| 후원자 환급 | ❌ | 구매대행이 됨 |

### 9-5. 잔액이 적법한 근거

잔액 자체는 위법이 아니다. 문제가 되는 것은 ① 고지 없는 용도 이탈 ② 무기한 방치 ③ 기록 부재 — 셋뿐이다.

**지정 범위를 물품이 아니라 시설 단위로 고지**하면 잔액 집행이 용도 이탈이 되지 않는다.

> 후원금은 **OO보육원**의 노트북 구입에 우선 사용되며, 잔액 발생 시 **같은 시설**의 다른 필요 물품에 사용됩니다.

이 고지가 후원 시점에 있어야 성립한다. 고지가 빠지면 아무리 잘 써도 용도 이탈이 된다.
단수 차액(수천 원 수준)은 회계상 정상이며, 이월 기록만 남기면 된다.

## 10. 운영 체계 (1인 운영 전제)

구매·배송 운영은 **관리자 1명으로 가능**하다. 단 아래 3가지가 시스템으로 뒷받침되어야 한다.

| 업무 | 주간 소요 |
|---|---|
| 마감 후 구매 (3건) | 10분 |
| 운송장 등록 | 5분 |
| 수령 확인 | 5분 |
| 다음 주 물품 선정·등록 | **1~2시간** ← 실제 병목 |
| 월 정산·회계 대사 | 월 2~3시간 |

**1) 시설이 직접 물품 신청 → 관리자는 승인만**
시설 계정으로 필요 물품(사진·사유·희망가)을 등록하고, 관리자는 검토·승인만 한다. 관리자 업무가 "조사"에서 "승인"으로 바뀌며 주 1~2시간 → 20분으로 줄고, 시설이 직접 쓴 사유가 후원 전환에도 유리하다. `facility` 도메인에 **시설 계정·권한 추가**가 필요하다.

**2) 라운드 자동 스킵**
승인된 물품이 3건 미만이면 그 주 라운드를 건너뛴다. 주간 고정 사이클이므로 이것이 없으면 관리자 휴가·병가에 서비스가 멈춘다.

**3) 수령 확인은 시설 권한**
`ItemStatus.RECEIVED` 전이 권한을 관리자가 아닌 **시설**에 준다. 1인이 물품 선정·구매·증빙 등록을 모두 통제하는 내부통제 공백을, 관리자가 단독으로 만들 수 없는 증거로 보완한다.

> **주의:** 운영의 진짜 병목은 구매가 아니라 **모금이 되게 만드는 일**(콘텐츠·시설 스토리·후기·홍보)이다. 1인 체제라면 여기에 시간을 배분해야 한다.

## 11. 코드 변경 요약

기존 도메인이 이미 흐름을 갖고 있어 구조 변경은 크지 않다.

**`DonationItem`** (`ItemStatus`: `RECRUITING → LOCKED → BUYING → SHIPPING → RECEIVED → RECEIPTED` 기존 유지)
- `FAILED` 상태 추가 (모금 미달)
- 라운드(주차) 식별자 + 마감 시각 (`deadline`이 `LocalDate`이므로 시각 포함 타입 검토)
- **`unitPrice` 추가** — 수량형 물품(9-1). `targetQuantity`는 기존 보유
- **최소/희망 목표** 2단 필드 — 수량형(9-2)
- `raisedAmount` → `Σ Contribution` 기준으로 전환 (기존 주석에 예고됨)
- 목표 달성 갱신 시 **동시성 락**(비관적 락 또는 `@Version`) — 초과 모금 방지

**`Contribution`** (`ContributionStatus`: 현재 `RECORDED`만 존재)
- `PENDING / PAID / FAILED / CANCELED / REFUNDED` 추가
- `paymentKey`(UNIQUE) · `pgProvider` · `paidAt` · `receiptUrl` 추가

**`PurchaseOrder`** (`courier` · `trackingNo` · `shippedAt` 기존 보유)
- `deliveryStatus` (`READY / IN_TRANSIT / OUT_FOR_DELIVERY / DELIVERED`) · `deliveredAt` 추가

**신규**
- 포인트 잔액 · 포인트 이력 (환급 전용)
- **시설별 잔액 풀** + 적립·집행 이력 (9-4) — 공개 화면 포함
- **시설 상시 소모품** 등록 (잔액 집행 대상)

- 결제 웹훅 엔드포인트 — **서명 검증 + 금액 대조 + 멱등성** 필수

- 일요일 마감 정산 배치 — 달성 판정 · 구매 수량 계산 · 잔액 적립 · 라운드 자동 스킵

**`notification`**
- 모금 완료 / 전달 완료 / 환급 이벤트 연결

## 12. 결정·확인이 남은 것

| 항목 | 담당 | 비고 |
|---|---|---|
| PG 선택 (포트원 vs 토스페이먼츠 직연동) | 기획 | 정기후원 계획 있으면 토스페이먼츠 |
| 사업자/법인 명의 확보 | 기획 | **PG 심사 1~2주 — 유일한 일정 병목** |
| 기부금영수증 발급 주체·자격 | 기획 + 세무 | 지정기부금단체 자격 여부 |
| 기부금품 모집 등록 필요 여부 | 기획 + 법무 | 모집비용 상한 규정 확인 |
| 시설 여러 곳일 때 노출 순번 규칙 | 기획 | |
| 라운드 초기 목표 금액대 | 기획 | 초반은 낮게 |
| 운영자금(cash float) 규모 | 기획 | 한 라운드분 |
| **DB 마이그레이션 도구(Flyway) 도입** | 개발 | 실결제 전 필수 — 아래 참고 |

### DB 스키마 — enum 체크 제약 주의

현재 `ddl-auto: update` 에 의존한다. 컬럼·테이블은 자동 추가되지만
**Hibernate 가 `@Enumerated(STRING)` 에 만든 체크 제약은 갱신되지 않는다.**
enum 값을 추가하면 런타임 INSERT 에서 `..._status_check` 위반으로 터진다.

값을 추가할 때마다 손으로 갱신해야 한다:

```sql
ALTER TABLE contributions DROP CONSTRAINT contributions_status_check;
ALTER TABLE contributions ADD CONSTRAINT contributions_status_check
  CHECK (status IN ('RECORDED','PENDING','PAID','FAILED','CANCELED','EXPIRED','REFUNDED'));
```

`ItemStatus` 에 `FAILED`(모금 미달)를 넣을 때 `donation_items_status_check` 에서 같은 문제가 재발한다.
**실결제 도입 전 Flyway 를 넣어 스키마를 코드와 함께 버전 관리**하는 것이 근본 해결이다.

> 법·세무 항목은 최종적으로 전문가 검토가 필요하다. 이 문서는 설계 방향을 고정하기 위한 정리이지 법률 자문이 아니다.
