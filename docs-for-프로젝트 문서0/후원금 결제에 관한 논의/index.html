단계별 MVP 전략 시설별 후원금 풀 자체 물류센터 법무 검토 필요 작성일 2026-07-19

후원금 결제와 포인트 기반 후원 설계 논의

초기에는 후원자가 외부 쇼핑몰에서 직접 구매한 물품을 자체 물류센터로 모으고, 플랫폼이 입고, 검수, 시설 전달, 증빙 공유를 맡는다. 이후 공동 현금 후원, 포인트 결제, 자체 쇼핑몰, 물류 자동화로 확장한다.

결론

현실적인 순서는 물류 검증을 먼저 하고, 결제/포인트/쇼핑몰 기능은 뒤로 미루는 것이다. MVP 1차는 자체 물류센터를 활용한 직접 구매 후원으로 시작하고, MVP 2차부터 공동 현금 후원과 플랫폼 구매를 단계적으로 붙인다.
MVP 1차 후원자가 직접 구매하고 영수증/송장을 첨부한다. 플랫폼은 물류센터 입고부터 시설 전달까지 관리한다.
MVP 2~4차 비싼 물품은 단체 현금 후원으로 공동 구매하고, 이후 자체 쇼핑몰 구매까지 확장한다.
MVP 5차 자체 쇼핑몰과 공동 구매에 물류센터 자동화, 입고 검수, 배송 추적 시스템을 결합한다.

현실적인 단계별 MVP 전략

단계 핵심 모델 후원자 역할 플랫폼 역할 검증 목표
MVP 1 자체 물류센터 활용 외부 쇼핑몰에서 물품을 직접 구매하고 영수증, 송장, 주문 내역을 첨부한다. 물류센터 주소 제공, 입고 확인, 검수, 분류, 시설 전달, 웹 공유를 맡는다. 시설 물품 공시, 후원자 직접 구매 의향, 입고/검수/전달 운영 가능성을 검증한다.
MVP 2 고가 물품 공동 현금 후원 노트북, 대형 가전, 계절 물품처럼 단가가 높은 물품에 금액으로 공동 후원한다. 공동 모금, 구매 집행, 영수증/송장 첨부, 시설 전달, 후원자 공지를 담당한다. 후원금 풀, 공동 구매, 차액 처리, 환불 정책, 구매 증빙 흐름을 검증한다.
MVP 3 자체 쇼핑몰 전환 플랫폼 안에서 시설별 필요 물품을 선택해 구매 또는 후원한다. 상품 등록, 주문, 구매, 배송, 시설 전달, 증빙 공개를 일괄 관리한다. 후원 UX를 단순화하고 외부 쇼핑몰/송장 첨부 부담을 줄인다.
MVP 4 하이브리드 자체 쇼핑몰에서 바로 구매하거나, 고가 물품은 공동 후원에 참여한다. 자체 쇼핑몰 주문과 MVP 2의 공동 구매를 함께 지원한다. 소액/단품 후원과 고가/공동 후원을 하나의 서비스 안에서 운영한다.
MVP 5 물류센터 자동화와 배송 추적 확장 입고, 검수, 배송, 전달 상태를 웹에서 추적한다. 입고 스캔, 송장 자동 매칭, CCTV/사진 증빙, 시설별 출고, 배송 추적을 고도화한다. 운영 비용을 낮추고 물류 신뢰성과 추적 가능성을 높인다.
이 순서는 금융 시스템을 먼저 크게 만들지 않고, 물품 후원 수요와 시설 전달 운영을 먼저 검증한다. 이후 결제와 자체 쇼핑몰을 붙이면 이미 검증된 물류/증빙 흐름 위에서 확장할 수 있다.

단계별 책임 경계

MVP 1의 책임 경계 후원자는 외부 구매와 영수증/송장 첨부까지 담당한다. 플랫폼은 물류센터 입고 이후의 검수, 보관, 시설 전달, 전달 증빙 공개를 책임진다.
MVP 2의 책임 경계 후원자는 금액 후원까지만 담당한다. 플랫폼 관리자가 공동 구매, 영수증, 송장, 전달 증빙, 후원자 메시지 발송을 처리한다.
MVP 3 이후 책임 경계 후원자는 플랫폼 안에서 구매/후원을 완료한다. 플랫폼은 쇼핑몰, 결제, 구매, 물류, 증빙을 하나의 운영 시스템으로 관리한다.

시설별 후원금 풀 후원금은 시설에 귀속되고, 공시된 물품 구매에 사용된다. 운영 복잡도를 줄이면서 투명성을 확보할 수 있다. 사후 매칭과 증빙 관리가 필요하다. MVP 권장

MVP 권장 흐름

1. 포인트 충전 후원자는 카드/간편결제로 후원 포인트를 충전한다.
2. 시설 선택 사전에 공시된 필요 물품 목록을 보고 시설에 후원한다.
3. 시설 후원금 풀 후원금은 시설별 사용 가능 잔액으로 이동한다.
4. 구매 실행 관리자가 공시 물품을 중심으로 실제 구매를 진행한다.
5. 사용 매칭 구매금액에 어떤 후원자의 금액이 쓰였는지 연결한다.
6. 증빙 공개 후원자에게 구매 내역, 영수증, 전달 증빙을 보여준다.

후원자에게 보여줄 표현

권장 문구 이 시설은 현재 겨울 내복, 학습 문제집, 위생용품 구매를 준비하고 있습니다. 후원금은 공개된 필요 물품 구매에 사용되며, 구매 완료 후 사용 내역과 증빙을 확인할 수 있습니다.
피해야 할 문구 이 물품을 직접 구매합니다. 이 노트북에 10,000원을 후원합니다. 목표 달성 시 반드시 이 상품을 구매합니다.
물품은 후원 대상이 아니라 구매 예정 항목으로 표현한다. 후원 대상은 시설이며, 물품은 사전 공시와 사후 증빙의 기준이다.

테이블 설계 원칙

  • 잔액은 직접 덮어쓰지 않고 거래 원장을 append-only로 기록한다.
  • PG 결제/환불 콜백은 idempotency_key와 provider transaction key로 중복 반영을 막는다.
  • 같은 포인트가 두 구매에 동시에 쓰이지 않도록 DB 트랜잭션과 version 기반 낙관적 락을 사용한다.
  • 구매 한 건에 어떤 후원자의 후원금이 얼마씩 쓰였는지 PURCHASE_FUND_USAGE로 남긴다.
  • 관리자 수동 조정은 AUDIT_LOG에 사유, 전후 값, 수행자를 기록한다.
  • 물품 가격 변동과 차액은 후원자별 즉시 환불보다 시설 후원금 풀 내 잔액 조정으로 우선 처리한다.

권장 ERD

상세 Mermaid ERD는 같은 폴더의 erd.mmd에 둔다. 핵심은 지갑 원장, 시설 후원금 원장, 구매 주문, 구매별 후원금 사용 매칭을 분리하는 것이다.

USER -> WALLET -> WALLET_TRANSACTION -> FACILITY_FUND_TRANSACTION -> PURCHASE_FUND_USAGE -> PURCHASE_ORDER -> DONATION_USAGE_REPORT

법무/정책 검토 포인트

이 문서는 제품/기술 설계 초안이며 법률 자문이 아니다. 포인트 충전, 미사용 잔액 환불, 선불충전금 별도관리, 전자금융거래법상 선불전자지급수단 해당 여부는 별도 법무 검토가 필요하다.
  • 충전 포인트가 선불전자지급수단에 해당하는지
  • 미사용 포인트 환불 가능 기준과 환불 수수료 정책
  • 시설 후원금 풀로 배정된 금액의 취소/환불 가능 시점
  • 구매 완료 후 환불 제한 근거와 고지 방식
  • 목표 물품 변경, 품절, 대체 구매 시 후원자 동의가 필요한 범위
  • 별도 계좌 관리 또는 선불충전금 별도관리 의무 적용 가능성
  • 후원금과 물품 구매대행, 기부금, 포인트 결제의 회계 처리 기준

다음 구현 순서

  1. 유저 관리, 역할 관리, 권한 관리, 역할-권한 매핑을 먼저 CRUD로 완성한다.
  2. 시설별 공시 물품 상태와 관리자 구매 워크플로우를 정리한다.
  3. 지갑/포인트 원장 테이블을 설계하고 결제 없이 내부 테스트 데이터를 만든다.
  4. PG 단건 충전을 연결하고 멱등성, 중복 콜백, 환불 테스트를 작성한다.
  5. 시설 후원금 풀 배정과 구매 사용 매칭을 구현한다.
  6. 후원자 사용 리포트와 증빙 화면을 만든다.
