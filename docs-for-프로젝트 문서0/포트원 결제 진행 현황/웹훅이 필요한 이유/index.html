웹훅이 필요한 이유

milla-pilot · 포트원 결제 연동 · 2026-07-21

한 줄 요약

지금 구조에서는 돈은 빠져나갔는데 후원은 실패하는 경우가 생긴다. 결제 성공을 후원자의 브라우저가 알려주기 때문이다. 브라우저가 그 일을 못 하면 아무도 알려주지 않는다.

문제 — 확정 신호가 브라우저에서 온다

후원자 → 카카오페이 결제 승인   ✅ 돈이 빠져나감
        ↓
      브라우저가 서버에 "확정해줘" 요청   ← 여기가 끊기면
        ↓
      서버가 포트원 조회 → PAID 확정      ← 영영 실행되지 않음

결제 승인은 카카오페이와 포트원 사이에서 끝난다. 우리 서버는 그 사실을 모른다. 알려주는 역할을 지금은 브라우저가 맡고 있는데, 브라우저는 언제든 사라질 수 있다.

결과 — 후원자는 결제했는데 15분 뒤 EXPIRED 로 정리된다. 모금액에 반영되지 않고, 참여 내역에도 남지 않는다. 후원자 입장에서는 돈만 사라진 것이다. 기부 서비스에서 이 한 번은 신뢰를 통째로 잃는다.

지금의 방어 — 3중이지만 전부 브라우저에 의존한다

경로동작못 막는 경우
① SDK 응답 결제창이 결과를 돌려주면 확정 요청 QR 결제·팝업 강제 종료 시 응답이 오지 않음
② 확정 폴링 3초마다 서버에 확정을 재시도 (최대 5분) 탭을 닫으면 폴링도 멈춤
③ 리다이렉트 복귀 모바일 복귀 시 URL 파라미터로 확정 복귀 전에 브라우저를 닫으면 끝
세 경로 모두 브라우저가 살아 있어야 동작한다. 결제 직후 탭을 닫거나, 지하철에서 통신이 끊기거나, 배터리가 나가면 전부 무력하다. 모바일 결제일수록 이 확률이 높다.

웹훅 — 포트원이 우리 서버에 직접 알려준다

후원자 → 카카오페이 결제 승인
        ↓
   포트원 ──────────────▶ 우리 서버   ← 브라우저와 무관
        POST /api/webhooks/portone
        ↓
   서버가 포트원 조회 → 금액 대조 → PAID 확정

후원자가 무엇을 하든, 브라우저가 살아 있든 아니든 서버 대 서버로 통보가 온다. 포트원은 응답이 없으면 여러 번 재시도하므로 일시적인 서버 장애도 넘긴다.

웹훅을 붙이면 위 3경로는 속도를 위한 보조가 된다. 후원자가 화면을 보고 있으면 즉시 완료 화면을 띄우고, 아니면 웹훅이 조용히 확정한다. 어느 쪽이든 후원은 반드시 기록된다.

구현 시 지켜야 할 것

  1. 서명 검증 — 웹훅 URL은 외부에 열려 있다. 검증 없이 받으면 누구나 "결제됐다"고 위조할 수 있다. 포트원 웹훅 시크릿으로 반드시 검증한다.
  2. 금액 재대조 — 웹훅 본문을 믿지 않고, 지금처럼 포트원에 다시 조회해 금액을 확인한다.
  3. 멱등성 — 재시도 때문에 같은 통보가 여러 번 온다. paymentId UNIQUE + 이미 PAID 면 무시하는 처리가 이미 되어 있다.
  4. 빠른 응답 — 무거운 작업을 웹훅 안에서 하면 타임아웃으로 재시도가 쌓인다. 확정만 하고 알림 발송 등은 분리한다.
  5. 로컬 수신 — 포트원은 localhost 로 못 보낸다. 개발 중에는 ngrok 같은 터널이 필요하다.

웹훅도 만능은 아니다 — 정산 대사가 최후 방어선

웹훅이 유실되는 경우도 드물게 있다. 서버가 오래 죽어 있었거나, 재시도 한도를 넘겼거나. 그래서 실서비스에서는 주기적인 대사(reconciliation) 를 함께 둔다. 

매일 1회 배치
  포트원의 PAID 목록  ↔  우리 DB의 PAID 목록
  → 포트원엔 있는데 우리에 없는 건 = 유실된 결제 → 복구
  → 우리에 있는데 포트원에 없는 건 = 이상 → 알림

웹훅 다음 단계로 두면 된다. 다만 실결제를 열기 전에는 웹훅까지가 필수다.
