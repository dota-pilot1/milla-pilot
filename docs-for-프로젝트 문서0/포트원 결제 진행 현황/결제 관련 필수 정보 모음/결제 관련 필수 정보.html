실연동 전환 방법

코드 수정 없이 환경변수만 바꾸면 된다. 채널 키는 application.yaml 에서 ${PORTONE_CHANNEL_KEY:테스트값} 으로 읽으므로, EC2 .env 에 한 줄을 추가하면 덮어쓴다. 

# /home/ubuntu/milla/.env 에 추가 후 재시작
PORTONE_CHANNEL_KEY=channel-key-520f32cd-acfc-4e1b-8909-86f6b5bfa66e
이 한 줄이 실제 돈이 오가기 시작하는 지점이다. 바꾸기 전에 아래 7절 체크리스트를 먼저 통과해야 하고, 전환 직후에는 소액 물품으로 1건만 결제해 전체 흐름(결제 → 확정 → 웹훅 → 목표 반영)을 확인한다.

Store ID 와 채널 키는 브라우저에 노출되는 공개 값이라 문서에 적어도 된다. 비밀값은 API Secret 과 웹훅 시크릿 둘뿐이다.

발급 위치

  • API Secret — 콘솔 → 결제 연동 → 식별코드 · API Keys → V2 API
  • 웹훅 시크릿 — 콘솔 → 결제 연동 → 결제알림(Webhook) 관리 → + 웹훅 시크릿 발급
둘 다 발급 순간에만 보인다. 목록에는 이름만 남고 값은 다시 볼 수 없다. 발급하면 바로 두 곳(로컬·EC2)에 넣는다. 다른 프로젝트(hibot)와 계정을 공유하므로 기존 키를 삭제하지 말고 추가 발급한다.

웹훅 버전결제모듈 V2 · 버전 2024-01-01 · application/json
재전송 정책200 이 아니면 최대 5회 · 0 → 1 → 4 → 16 → 64 → 256분
서명 헤더webhook-id · webhook-timestamp · webhook-signature (Standard Webhooks)
서명 방식HMAC-SHA256({id}.{timestamp}.{본문}) · 키는 whsec_ 제거 후 base64 디코드 · 타임스탬프 5분 허용
시크릿이 비어 있으면 모든 웹훅을 401 로 거절한다(fail-closed). 검증할 수 없는 요청을 받아주면 누구나 "결제됐다"고 위조할 수 있기 때문이다. 웹훅이 안 들어오면 가장 먼저 시크릿 설정을 의심한다.

enum 체크 제약 갱신 — enum 값을 추가할 때마다 필요

ALTER TABLE contributions DROP CONSTRAINT contributions_status_check;
ALTER TABLE contributions ADD CONSTRAINT contributions_status_check
  CHECK (status IN ('RECORDED','PENDING','PAID','FAILED','CANCELED','EXPIRED','REFUNDED'));

7. 실결제 전환 체크리스트

  1. PG 계약 완료 · 계약 명의가 단체/법인인지 확인 — 개인 명의는 소득·투명성 문제가 생긴다
  2. 기부금품 모집 등록 필요 여부 확인 (법무 — 사업 형태가 바뀔 수 있다)
  3. 기부금영수증 발급 주체·자격 확정 (세무)
  4. 계약서에서 수수료율 · 정산주기(D+N) 확인 → 목표액 산정식과 운영자금 규모에 반영
  5. 웹훅을 실연동 모드로도 등록
  6. 실연동 채널 키로 교체 — 환경변수만 변경
  7. 소액 실결제 1회 검증 — 목업과 실제는 앱 전환·복귀 흐름이 다르다
  8. 정산 대사 배치 도입 검토 (웹훅 유실 대비 최후 방어선)
  9. 포트원 상점을 프로젝트별로 분리 (현재 hibot 과 공유 — 기부금 회계는 분리가 중요)
