지금 어디까지 왔나

테스트 채널로 결제가 완전히 동작하고, 운영에도 배포됐다. 결제 요청 → 카카오페이 승인 → 서버가 포트원에 직접 조회해 금액 대조 → 후원 확정 → 목표 달성 시 잠금까지 실제 결제 6건으로 검증했다. 웹훅도 붙어서 포트원 호출을 수신하고 서명 검증을 통과하는 것까지 확인했다.

아직 실결제는 열려 있지 않다. 실연동 채널 키로 바꾸지 않았기 때문이다. 그 전에 아래 TODO 의 법무·세무 항목이 먼저 정리돼야 한다.

실결제 검증을 30만원짜리 물품으로 할 수는 없다. 시더에 검증 전용 시설을 추가했다.

시설 코드verify · 이름 [검증] 결제 테스트 · displayOrder 99(목록 맨 뒤)
물품결제 검증용 A — 목표 1,000원 / 결제 검증용 B — 목표 500원
후원 금액프리셋(1만·3만·5만)은 남은 금액 초과 시 자동 비활성. "남은 금액 모두"와 직접 입력으로 소액 결제 가능
남은 일 — 배포해야 시드가 반영된다. 또한 운영 공개 전에는 이 시설을 숨기거나 제외해야 한다. 기존 시설의 목표액은 일부러 현실 금액(30~60만원)으로 유지했다. 시연·기획 검토용 자산이라 "노트북 3대 1,000원"은 설득력이 없기 때문이다.

TODO 1.5 — 만료 배치가 실결제 건을 지운다 ✅ 코드 완료

(수정 전) expirePendings() 가 포트원에 조회하지 않고 만료된 PENDING 을 그대로 EXPIRED 로 바꿨다. 결제 승인 후 브라우저 닫힘과 웹훅 유실이 겹치면 후원자가 돈을 냈는데 시스템상 "결제 안 한 사람"이 된다. PENDING 으로 남아만 있으면 나중에 대사해 되살릴 수 있는데, EXPIRED 라는 종료 상태로 확정해 회복 경로까지 닫아버렸다.

적용한 원칙 — 확실히 아닌 것만 만료시킨다

포트원 조회 결과처리
404 (기록 없음)결제창까지 안 감 → EXPIRED. 만료해도 안전한 유일한 경우
PAID · 금액 일치후원 확정(RESCUED) — 원래 잃던 건을 되살린다
PAID · 금액 불일치FAILED + error 로그. 자동 확정하지 않고 사람이 본다
FAILED/CANCELLEDEXPIRED
READY 등 승인 전보류 — PENDING 유지, 다음 주기 재확인. PORTONE_PENDING_HARD_EXPIRE_HOURS(기본 24h) 넘으면 만료
조회 실패(장애·401)보류 — 만료시키지 않고 다음 주기 재시도

보류해도 안전한 근거 — 만료 시각이 지난 PENDING 은 sumHeldAmount 에서 이미 제외돼 잔여 금액을 선점하지 않는다. 남겨둬도 다른 후원자를 막지 않는다.

변경 파일

  • PortOneClient — findPayment() 추가. 404(기록 없음)와 조회 실패를 구분한다. 전자는 만료해도 되지만 후자에서 만료하면 결제된 건을 잃는다
  • ContributionService — settleExpiredPending(id) 건별 트랜잭션. applyVerification() 을 분리해 확정 경로와 같은 검증 규칙(상태·금액 대조)을 공유
  • PaymentExpiryScheduler — 루프·건별 예외 격리·요약 로그. 한 건의 조회 실패가 나머지를 롤백시키지 않도록 루프를 서비스 밖에 뒀다
  • ContributionRepository.findExpiredPendingIds(), application.yaml 설정 1개
남은 일 — 배포. 아직 운영에 반영되지 않았다. 실연동 전환(TODO 2) 전에 배포돼 있어야 한다.
전 기간 정산 대사 배치(포트원 PAID 목록 ↔ DB 전수 대조)는 나중에. 급한 구멍은 이걸로 막혔다. 주 3건 라운드 규모면 당분간 포트원 콘솔에서 월 1회 눈으로 맞춰도 충분하다.

TODO 3 — 계약 · 법무 · 세무 가장 큰 병목

개발은 거의 끝났고, 여기가 실제 일정을 결정한다. 순서대로 확인한다.

① PG 계약 명의 — 가장 먼저

  • 포트원 계약이 사업자/법인 명의인지 개인 명의인지 확인
  • 개인 명의면 후원금이 개인 소득으로 잡히고, 횡령 의혹에 구조적으로 노출된다
  • 확인 방법 — 포트원 콘솔 상점 관리 또는 계약서의 사업자등록번호

② 기부금품 모집 등록 필요 여부 — 사업 형태를 바꿀 수 있음

  • 「기부금품의 모집 및 사용에 관한 법률」상 일정 규모 이상 모집은 사전 등록이 필요하고, 미등록 모집은 형사처벌 대상이다
  • 주 3건 라운드에 고가 물품이 들어가면 기준을 빠르게 넘길 수 있다
  • 모집·운영비 사용 비율에 법적 상한도 있다 — 창고·인건비 계획을 접은 이유
  • 확인처 — 관할 지자체(시·도) 또는 행정안전부. 세무사·법무 상담 권장

③ 기부금영수증 발급 주체

  • 시설이 지정기부금단체인지, 아니면 우리가 자격을 받아야 하는지
  • 시설 명의로 발급 가능하면 회계·세무가 가장 깔끔해진다
  • 자격 없이 발급하면 가산세

④ 계약 조건 숫자

수수료율목표액 = 구매가 + 수수료 + 여유 10% — 공식에 들어간다
정산주기 D+N모금 완료 시점에 현금이 없다. 운영자금(cash float) 규모를 결정
비영리·기부 우대정률이라 장기 비용 차이가 크다. 반드시 문의
법·세무 항목은 최종적으로 전문가 검토가 필요하다. 이 문서는 상담 전에 질문을 정리하기 위한 것이지 법률 자문이 아니다.

알려진 한계

  • 결제 직후 브라우저를 닫으면 확정 경로 3개가 모두 무력하다. 웹훅이 이를 막지만, 웹훅 자체도 유실될 수 있다. 이 구멍은 TODO 1.5 로 막혔다 — 만료 배치가 포트원에 직접 확인해 실제 결제된 건은 되살리고, 확실하지 않으면 만료시키지 않는다. 전 기간 정산 대사는 그 뒤의 최후 방어선이며 아직 없다
  • 확정 실패 사유가 저장되지 않는다 — 예외를 던지면 트랜잭션이 롤백되어 failReason 기록도 사라진다. 운영 진단이 필요해지면 별도 트랜잭션으로 분리
  • 실연동 흐름은 미검증 — 테스트는 목업이라 앱 전환·복귀가 실제와 다르다
  • 발주는 수동 — 소비자 대신 주문하는 API 는 국내 대형 커머스에 없다. 주 1회 5분 수준
