지금 어디까지 왔나

테스트 채널로 결제가 완전히 동작하고, 운영에도 배포됐다. 결제 요청 → 카카오페이 승인 → 서버가 포트원에 직접 조회해 금액 대조 → 후원 확정 → 목표 달성 시 잠금까지 실제 결제 6건으로 검증했다. 웹훅도 붙어서 포트원 호출을 수신하고 서명 검증을 통과하는 것까지 확인했다.

아직 실결제는 열려 있지 않다. 실연동 채널 키로 바꾸지 않았기 때문이다. 그 전에 아래 TODO 의 법무·세무 항목이 먼저 정리돼야 한다.

TODO 1 — 소액 검증 준비 ✅ 코드 완료

실결제 검증을 30만원짜리 물품으로 할 수는 없다. 시더에 검증 전용 시설을 추가했다.

시설 코드verify · 이름 [검증] 결제 테스트 · displayOrder 99(목록 맨 뒤)
물품결제 검증용 A — 목표 1,000원 / 결제 검증용 B — 목표 500원
후원 금액프리셋(1만·3만·5만)은 남은 금액 초과 시 자동 비활성. "남은 금액 모두"와 직접 입력으로 소액 결제 가능
남은 일 — 배포해야 시드가 반영된다. 또한 운영 공개 전에는 이 시설을 숨기거나 제외해야 한다. 기존 시설의 목표액은 일부러 현실 금액(30~60만원)으로 유지했다. 시연·기획 검토용 자산이라 "노트북 3대 1,000원"은 설득력이 없기 때문이다.

전환 직후 검증 절차

  1. [검증] 결제 테스트 시설 → 결제 검증용 B(500원) 후원
  2. 완료 화면이 뜨고 목표 달성 → LOCKED 로 넘어가는지 확인
  3. 서버 로그에서 웹훅 수신 확인 
    sudo journalctl -u milla-server --since '5 min ago' --no-pager | grep webhook
  4. 포트원 콘솔 결제 내역 에서 실제 승인 확인 → 바로 취소
이 한 줄이 실제 돈이 오가기 시작하는 지점이다. TODO 3 이 정리되기 전에는 바꾸지 않는다. 결제는 기술이 아니라 계약과 법적 자격의 문제다.

TODO 3 — 계약 · 법무 · 세무 가장 큰 병목

개발은 거의 끝났고, 여기가 실제 일정을 결정한다. 순서대로 확인한다.

① PG 계약 명의 — 가장 먼저

  • 포트원 계약이 사업자/법인 명의인지 개인 명의인지 확인
  • 개인 명의면 후원금이 개인 소득으로 잡히고, 횡령 의혹에 구조적으로 노출된다
  • 확인 방법 — 포트원 콘솔 상점 관리 또는 계약서의 사업자등록번호

② 기부금품 모집 등록 필요 여부 — 사업 형태를 바꿀 수 있음

  • 「기부금품의 모집 및 사용에 관한 법률」상 일정 규모 이상 모집은 사전 등록이 필요하고, 미등록 모집은 형사처벌 대상이다
  • 주 3건 라운드에 고가 물품이 들어가면 기준을 빠르게 넘길 수 있다
  • 모집·운영비 사용 비율에 법적 상한도 있다 — 창고·인건비 계획을 접은 이유
  • 확인처 — 관할 지자체(시·도) 또는 행정안전부. 세무사·법무 상담 권장

③ 기부금영수증 발급 주체

  • 시설이 지정기부금단체인지, 아니면 우리가 자격을 받아야 하는지
  • 시설 명의로 발급 가능하면 회계·세무가 가장 깔끔해진다
  • 자격 없이 발급하면 가산세

④ 계약 조건 숫자

수수료율목표액 = 구매가 + 수수료 + 여유 10% — 공식에 들어간다
정산주기 D+N모금 완료 시점에 현금이 없다. 운영자금(cash float) 규모를 결정
비영리·기부 우대정률이라 장기 비용 차이가 크다. 반드시 문의
법·세무 항목은 최종적으로 전문가 검토가 필요하다. 이 문서는 상담 전에 질문을 정리하기 위한 것이지 법률 자문이 아니다.

알려진 한계

  • 결제 직후 브라우저를 닫으면 확정 경로 3개가 모두 무력하다. 웹훅이 이를 막지만, 웹훅 자체도 유실될 수 있어 정산 대사가 최후 방어선이다
  • 확정 실패 사유가 저장되지 않는다 — 예외를 던지면 트랜잭션이 롤백되어 failReason 기록도 사라진다. 운영 진단이 필요해지면 별도 트랜잭션으로 분리
  • 실연동 흐름은 미검증 — 테스트는 목업이라 앱 전환·복귀가 실제와 다르다
  • 발주는 수동 — 소비자 대신 주문하는 API 는 국내 대형 커머스에 없다. 주 1회 5분 수준
