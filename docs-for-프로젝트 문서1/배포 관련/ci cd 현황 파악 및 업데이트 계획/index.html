배포 문서 CI active Deploy active 배포는 수동 실행

CI/CD 현황 파악 및 업데이트 계획

밀라 프로젝트는 Spring 백엔드, donor-facing Next.js 웹, 관리자 Tauri 앱이 분리되어 있다. 2026-07-18 기준으로 repo 루트에 CI와 Deploy GitHub Actions workflow를 추가했다. CI는 push/PR에서 자동 실행되고, Deploy는 GitHub Actions 화면에서 수동 실행한다. 첫 CI에서 발견된 Next.js 정적 export 문제는 라우트 전환 후 해결되어 최신 CI가 통과했다.

1. 결론

1차 개선 완료. 이전에는 자동 검증/배포 workflow가 없었지만, 현재는 .github/workflows/ci.yml과 .github/workflows/deploy.yml이 main에 반영되어 GitHub Actions에서 활성화되어 있다. 운영 배포는 사고 방지를 위해 자동 push 배포가 아니라 workflow_dispatch 수동 실행으로 둔다.

3. 왜 이렇게 했는가

1) 프로젝트가 단일 앱이 아니다

이 repo는 백엔드, 웹, 관리자 앱이 함께 움직인다. 한쪽 타입·API·빌드가 깨져도 다른 쪽에서 바로 눈치채기 어렵다. 자동 CI가 없으면 “내 로컬에서는 됨” 상태로 main이 깨질 가능성이 높다.

2) 배포 대상이 이미 운영형 구조다

EC2, systemd, S3, CloudFront, Tauri updater 문서가 이미 있다. 이 정도 인프라를 수동으로만 관리하면 배포 재현성과 롤백 판단이 사람 기억에 의존한다.

3) 후원 도메인은 신뢰가 중요하다

결제 전 단계의 MVP라도 사용자·시설·후원 이력·비밀번호 재설정 같은 민감한 흐름이 있다. 최소한 빌드 실패, 보안 설정 누락, API 계약 깨짐은 merge 전에 잡아야 한다.

4) 수동 배포는 속도보다 사고 비용이 커진다

지금은 기능 추가 속도가 빠르기 때문에 수동 체크리스트가 누락되기 쉽다. CI는 개발 속도를 늦추는 장치가 아니라, 같은 실수를 반복해서 확인하지 않게 만드는 기본 장치다.

4. 현재 CI 구성

대상 현재 명령 CI에서 할 일 비고
백엔드 cd donation-platform-server && ./gradlew test Java 21 toolchain으로 테스트 실행 서버를 띄우지 않는다. 테스트만 수행.
웹 cd donation-platform-front && npm ci && npm run lint && npm run build Next.js lint/build 확인 NEXT_PUBLIC_API_URL 기본값이 localhost라도 정적 빌드가 되는지 확인.
관리자 cd donation-admin-tauri && npm ci && npm run build TypeScript + Vite build 확인 Tauri native packaging은 별도 release 단계로 분리.
핵심: 처음부터 서버 배포, 프론트 배포, Tauri 릴리스까지 한 번에 자동화하지 않는다. 먼저 PR에서 “빌드 가능한 코드인가”를 보장하고, 그 다음 main 배포를 붙이는 순서가 안정적이다.

5. 단계별 업데이트 현황

단계 작업 목적 완료 기준
1단계 루트 .github/workflows/ci.yml 추가 PR/Push 시 백엔드·웹·관리자 빌드 검증 완료 · 최신 CI 통과
2단계 path filter 적용 변경된 영역만 CI 실행해 시간 절약 미적용 · 현재는 세 job 모두 실행
3단계 브랜치 보호 설정 CI 실패 코드가 main에 들어가는 것 차단 GitHub Settings에서 별도 설정 필요
4단계 백엔드 main 배포 workflow 추가 JAR 빌드, EC2 업로드, systemd restart 재현성 확보 workflow 추가 · 실제 실행은 아직 별도
5단계 웹 main 배포 workflow 추가 Next 정적 빌드, S3 sync, CloudFront invalidation 자동화 workflow 추가 · 실제 실행은 아직 별도
6단계 Tauri release 전략 확정 관리자 앱 자동 업데이트 체계 정리 독립 repo 또는 모노레포 release 방식 중 하나로 고정

6. 현재 CI workflow

현재 .github/workflows/ci.yml의 구조는 아래와 같다. push/PR/수동 실행을 지원한다. 

name: CI

on:
  pull_request:
  push:
    branches: [main]
  workflow_dispatch:

jobs:
  backend:
    runs-on: ubuntu-latest
    defaults:
      run:
        working-directory: donation-platform-server
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-java@v4
        with:
          distribution: temurin
          java-version: "21"
      - run: chmod +x ./gradlew
      - run: ./gradlew test

  web:
    runs-on: ubuntu-latest
    defaults:
      run:
        working-directory: donation-platform-front
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 22
          cache: npm
          cache-dependency-path: donation-platform-front/package-lock.json
      - run: npm ci
      - run: npm run lint
      - run: npm run build

  admin:
    runs-on: ubuntu-latest
    defaults:
      run:
        working-directory: donation-admin-tauri
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 22
          cache: npm
          cache-dependency-path: donation-admin-tauri/package-lock.json
      - run: npm ci
      - run: npm run build
주의: 이 workflow는 검증용이다. 운영 서버 재시작, S3 sync, CloudFront invalidation은 Deploy workflow에서 수동 실행한다.

7. 현재 Deploy workflow

.github/workflows/deploy.yml은 자동 배포가 아니라 GitHub Actions 화면에서 직접 실행한다. 실행 시 target을 all, backend, frontend 중 선택한다.

target 동작 검증
backend Gradle build → JAR 백업/업로드 → milla-server restart /api/site-settings smoke check
frontend Next 정적 build → S3 sync → CloudFront invalidation 운영 루트 URL smoke check
all backend와 frontend job을 함께 실행 각 job smoke check
운영 기준: push 즉시 자동 배포하지 않는다. CI가 성공한 commit을 기준으로 사람이 Actions에서 Deploy를 눌러 배포한다.

8. 왜 자동 push 배포로 두지 않았는가

  • 현재 repo에는 이미 수동 배포 문서가 있어, 당장 배포 방법이 없는 상태는 아니다.
  • CD는 AWS 권한, SSH 키, 운영 env, CloudFront ID, systemd 권한을 다루므로 실수 비용이 CI보다 크다.
  • CI가 먼저 안정되어야 “배포해도 되는 아티팩트”와 “배포하면 안 되는 코드”를 구분할 수 있다.
  • 관리자 Tauri 앱은 updater 서명, GitHub Release, 앱별 repo 전략이 걸려 있어 웹/서버 CD와 분리해야 한다.

9. 등록된 GitHub Actions Secrets

영역 Secret 예시 용도
백엔드 EC2 EC2_HOST, EC2_USER, EC2_SSH_KEY JAR 업로드와 milla-server 재시작
프론트 AWS AWS_ACCESS_KEY_ID, AWS_SECRET_ACCESS_KEY, AWS_REGION S3 sync, CloudFront invalidation
프론트 배포 대상 FRONT_S3_BUCKET, CLOUDFRONT_DISTRIBUTION_ID, NEXT_PUBLIC_API_URL 정적 파일 업로드와 API URL 주입
Tauri updater TAURI_SIGNING_PRIVATE_KEY, TAURI_SIGNING_PRIVATE_KEY_PASSWORD 자동 업데이트 아티팩트 서명
macOS 공증 APPLE_CERTIFICATE, APPLE_ID, APPLE_PASSWORD, APPLE_TEAM_ID 외부 배포용 macOS 신뢰 경고 최소화
원칙: 운영 비밀값은 문서나 git에 쓰지 않는다. 현재 실제 값은 GitHub Actions Secrets에 등록되어 있으며, 문서에는 이름과 용도만 남긴다. 자세한 복붙용 안내는 배포 자동화 관련 깃허브 액션 변수 문서를 참고한다.

10. 권장 운영 기준

  1. PR은 CI 통과 전 merge하지 않는다.
  2. main push 배포는 초기에는 자동 실행보다 workflow_dispatch 수동 승인으로 시작한다.
  3. 백엔드 배포 전 ./gradlew test, 프론트 배포 전 npm run build 산출물을 기준으로 한다.
  4. 배포 후 smoke check는 최소 /api/site-settings HTTP 200과 프론트 루트 HTTP 200을 확인한다.
  5. 롤백은 “이전 JAR 보관”과 “이전 S3 artifact 보관”이 생긴 뒤 자동화한다.
