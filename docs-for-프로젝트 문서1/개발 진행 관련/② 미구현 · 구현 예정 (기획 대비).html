미구현 · 구현 예정 (기획 대비)

기획 문서가 요구하지만 아직 코드에 없는 것을 성격별로 분류한다. 이미 된 것은 ① 구현 완료 (기획 대비).html 참고. 분류: 본체 미완결 → 프로덕션 필수 → 운영 보강 → 순수 부가(차기).

요약

#항목분류우선현재 상태
1영수증 발급 단계(RECEIPTED)본체P0enum만, 전환·발급·노출 없음
2실결제(PG)프로덕션 필수P0RECORDED 기록만, 돈 이동 없음
3후원 취소·환불 정합성프로덕션 필수P0취소 상태·롤백 없음
4관리자 권한 세분화(permission)프로덕션 필수P1role 단위만, permission 미활용
5상태 전환 감사 로그운영 보강P1who/when 이력 없음
6후원자 알림운영 보강P1pull만, push 없음
7마감일(deadline) 만료 처리운영 보강P2필드만, 만료 로직 없음
8실구매액↔목표액 차액 정산운영 보강P2잉여/부족 규칙 미정
9인증·인가 견고화운영 보강P2거부 시 403 대신 401
10배송 자동추적 · 커뮤니티 · 통계 · i18n …순수 부가차기파일럿 이후
P0(본체·핵심): #1 루프 완결 → #2·#3 실돈 흐름. 실결제·환불은 전문가 검토 관문(§5·F-016)이 코드 이전 선행 조건.

A. 본체 미완결

본체 P0 1. 영수증 발급 단계(RECEIPTED)

기획
이행 루프 마지막 칸 "영수증 발급자료 → 후원자 화면 단계 반영"(§5·§8)
현재
ItemStatus.RECEIPTED·메뉴(ADMIN_RECEIPTS·/receipts)는 있으나 markReceipted() 전환 메서드·발급 도메인·후원자 노출 전무. RECEIVED에서 멈춤.
남은 일
수령확인 후 증빙 묶음(구매영수증·배송완료·수령확인)을 발급자료로 정리 → 웹 구매현황/후원내역에 "영수증" 노출. 세액공제용 아님 유지.

B. 프로덕션 필수 (실서비스 전 반드시)

근거: 제품 핵심 기능 및 범위.html §3 "프로덕션 필수 부가 — 본체는 아니지만 실서비스엔 필수"

프로덕션 P0 2. 실결제(PG)

기획
정책 "결제 기능은 전문가 검토 후 별도 단계". ContributionStatus 주석: "차기 PAID 실결제 — 관문 통과 후".
현재
공동충당은 RECORDED(가상 리허설) — 실제 돈 이동 없음. 목표달성도 "기록상".
남은 일
PG 선정(단건) → PAID + 결제키 저장. 후원금=시설 전용계좌 직접 귀속(정책) 반영. 관문 선행.

프로덕션 P0 3. 후원 취소 · 환불 정합성

기획
ContributionStatus 주석 "차기 REFUNDED". §5 정합성의 역방향.
현재
취소/환불 경로 없음. 상태값 RECORDED 하나뿐, raisedAmount 롤백·잠금 후 환불 규칙 미정.
남은 일
취소는 구매(BUYING) 전까지 허용을 기본선으로 — 비관락 차감 + 미달 시 RECRUITING 복귀. 구매 후 환불은 별도 정책.

프로덕션 P1 4. 관리자 권한 세분화(permission authz)

기획
§3 "관리자 authz 세분화 = 프로덕션 필수". permission·역할-권한 도메인은 이미 설계됨.
현재
SecurityConfig는 hasAnyRole 역할 단위만. Permission/매핑 도메인은 있으나 authz 미활용, @PreAuthorize 거의 없음.
남은 일
권한 코드(PURCHASE_EXECUTE·FACILITY_WRITE 등) 정의 → @PreAuthorize("hasAuthority(...)"). 기존 매핑 화면 재활용.

C. 운영 보강

운영 P1 5. 상태 전환 감사 로그 · 이력

기획
§3 "감사메시지 = 프로덕션 필수". 정책 "신뢰·운영 가능성 우선".
현재
물품 상태 전환(LOCKED→BUYING→SHIPPING→RECEIVED)에 누가·언제·왜 이력 없음(updatedAt만).
남은 일
StatusHistory(대상·이전→이후·actor·사유·시각) 추가 → 전환 시 append, 타우리 타임라인 조회.

운영 P1 6. 후원자 알림(단계 이벤트 통지)

기획
루프 종단 "후원자 화면에 단계 반영"(§8) — 현재는 재방문(pull)만.
현재
목표달성·구매·배송·수령 시 후원자에게 알리는 채널 없음.
남은 일
MVP는 인앱 알림함(이벤트 저장+뱃지)부터. 이메일/푸시는 이후.

운영 P2 7. 마감일(deadline) 만료 · 목표미달 처리

기획
물품 정책 "…목표금액, 마감일을 관리한다".
현재
DonationItem.deadline 필드만. 만료 자동처리·미달 마감 규칙 없음 — 지난 물품도 계속 모집중.
남은 일
경과 스케줄러/조회 파생상태 → 미달이면 EXPIRED + 후원분 환불(#3) 연동.

운영 P2 8. 실구매액 ↔ 목표액 차액 정산

기획
물품 정책 "목표금액은 실제 구매 총액 기준".
현재
purchasedAmount를 목표액과 다르게 기록 가능하나 잉여/부족 차액 처리·표시 미정.
남은 일
차액 정책(잔액=시설 후원금 귀속 등) + 구매현황에 "목표 vs 실구매·잔액" 표기. 실결제(#2) 붙으면 즉시 이슈화.

운영 P2 9. 인증 · 인가 견고화

기획
§3 authz·감사 위생의 하부.
현재
권한 거부 시 403 대신 401 등 상태코드 미세 결함(기능 차단은 정상). 리프레시 실패·연속실패 잠금 UX 얕음.
남은 일
인증실패=401 / 권한부족=403 정합화, 리프레시·잠금 정비.
