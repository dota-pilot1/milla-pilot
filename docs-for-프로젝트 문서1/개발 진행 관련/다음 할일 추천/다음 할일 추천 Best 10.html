Step 999 · 계획 다음 할일 추천 착수 순서 · 구현 여부

다음 할일 추천 — Best 10

① 구현 완료·② 미구현 · 구현 예정을 근거로 한 10가지 할일. 순번은 정책 우선순위가 아니라 "지금 바로 착수 가능 + 가시성" 기준 착수 순서로 재정렬했다 (정책 우선순위 P0·P1·P2는 배지로 그대로 표시). 각 항목에 현재 · 할 일 · 완료 기준(DoD) · 규모 · 구현 여부를 붙였다. 구현 여부 칸(⬜ 미구현 / 🚧 구현 중 / ✅ 완료)으로 진행을 추적한다.

한눈에 — 착수 순서 10

#항목분류 · 우선순위영역구현 여부착수
1후원자 알림(인앱 알림함)운영 P1BE WEB⬜ 미구현지금
2통계 대시보드차기 · 부가WEB APP✅ 완료 (웹 v1)—
3실결제(PG)프로덕션 P0BE WEB⬜ 미구현관문 대기
4영수증 발급 단계(RECEIPTED)본체 P0BE APP WEB⬜ 미구현지금
5후원 취소 · 환불(구매 전 취소)프로덕션 P0BE WEB⬜ 미구현정책 후
6관리자 권한 세분화(permission authz)프로덕션 P1BE⬜ 미구현지금
7상태 전환 감사 로그 · 이력운영 P1BE APP⬜ 미구현지금
8마감일(deadline) 만료 · 목표미달 처리운영 P2BE⬜ 미구현#5 이후
9실구매액 ↔ 목표액 차액 정산운영 P2BE WEB⬜ 미구현#3 이후
10인증 · 인가 견고화(401/403)운영 P2BE⬜ 미구현지금 (소)
순번 = 착수 순서(즉시 구현 가능성 + 가시성). 정책상 필수도(P0>P1>P2)와는 다르다 — 알림·대시보드는 정책 우선순위가 높진 않지만 선행 결정·관문 없이 오늘 바로 만들고 눈에 보이는 순수 부가라 맨 앞에 뒀다. 본체·프로덕션 P0(영수증·실결제·취소)는 정책상 더 중요하나, 각각 도메인 무게·전문가 관문·정책 확정이 걸려 뒤로 배치했다.

재정렬 근거 · 관문

지금 바로(결정·관문 불필요) — #1 알림 · #2 대시보드 · #4 영수증 · #6 authz · #7 감사 · #10 인증정합화
선행조건 있음 — #5 취소(취소 허용 범위 정책 확정 후) · #8 마감일(#5 롤백 연동) · #9 차액(#3 실결제 이후)
관문 대기 — #3 실결제(PG)
#3 실결제는 코드보다 관문이 먼저. 정책·§5·F-016 "결제·환불은 전문가 검토 후 별도 단계". 실결제와 그에 딸린 실환불·차액 정산(#9)은 관문 통과 전까지 착수 대상이 아니다. (취소의 구매 전 롤백(#5)은 실결제 없이 가능해 먼저 다룬다.)

항목별 착수 가이드

1. 후원자 알림(인앱 알림함) 운영 P1 ⬜ 미구현 BE WEB

왜 앞선행 결정·관문 없는 순수 additive. 루프 종단 "후원자 화면에 단계 반영"(§8) — 현재는 재방문(pull)만.
현재서버엔 EmailNotificationService(비번 재설정 메일)뿐, 인앱 알림 도메인 없음. 목표달성·구매·배송·수령 시 후원자 통지 채널 부재.
할 일
  • BE(신규): Notification 엔티티 + 상태 전환 시 후원자에게 emit + GET /api/notifications/me + 읽음 처리
  • WEB: 헤더 종 아이콘 뱃지 + 알림 목록 드롭다운/알림함 (표시는 후원자=웹)
완료 기준내가 후원한 물품이 배송중→수령으로 바뀌면 웹 알림함에 뱃지 + 항목이 뜬다.
규모중 — BE 도메인 신규 + WEB 알림함.

2. 통계 대시보드 차기 · 부가 ✅ 완료 (웹 v1) WEB APP

왜 앞가장 만만함 — 웹 /dashboard 페이지·메뉴가 이미 존재, 집계 소스(후원·물품·시설) API도 다 있음. 관문·정책·도메인 신규 무관. 데모 임팩트 큼.
완료(웹 v1)신규 API 없이 GET /api/contributions/me 하나로 클라이언트 집계. src/app/dashboard/page.tsx:
  • 요약 지표 4종 — 내 총 후원액 · 후원 건수 · 참여 물품 · 참여 시설
  • 물품 진행 상태 분포(Badge 카운트) · 참여 물품별 진행률(Progress) · 시설별 내 후원
  • 로딩 스켈레톤 · 빈 상태 · 에러 상태. 반응형은 자연 접힘 grid만(브레이크포인트 최소화)
기존 공용 UI(Card·Progress·Badge·PageHeader·EmptyState) 재사용. TypeScript 검사 통과.
남은(차기)운영 전역 통계(관리자용 집계 API·타우리 "운영 현황" 강화)는 데이터 축적 후. 지금은 후원자 개인 대시보드 범위.
규모완료 — 소(대충 v1).

3. 실결제(PG) 프로덕션 P0 ⬜ 미구현 ⚠ 관문 대기 BE WEB

왜 대기정책상 "결제는 전문가 검토 후 별도 단계"(§5·F-016). 관문 통과가 코드 착수의 선행 조건.
현재공동충당은 RECORDED(가상 리허설) — 실제 돈 이동 없음. 목표달성도 "기록상".
할 일 (관문 후)PG 선정(단건) → PAID + 결제키 저장. 후원금 = 시설 전용계좌 직접 귀속(정책) 반영.
완료 기준실제 결제 승인 → PAID 기록 + 시설 계좌 귀속 확인.
규모대 — 외부 PG 연동 + 정산/보안. 관문·법무 검토 포함.

4. 영수증 발급 단계(RECEIPTED) 본체 P0 ⬜ 미구현 BE APP WEB

왜 지금이행 루프의 마지막 칸. 본체 6/6 → 7/7 완결. 외부 관문 없음.
현재ItemStatus.RECEIPTED·메뉴(ADMIN_RECEIPTS·/receipts)는 있으나 markReceipted() 전환·발급 도메인·후원자 노출 전무. RECEIVED에서 멈춤.
할 일
  • BE: DonationItem.markReceipted() + 전환 API, 증빙 묶음(구매영수증+배송완료+수령확인) 발급
  • APP: 타우리 /receipts 스텁 → 실화면 · WEB: 구매현황/후원내역에 "영수증" 노출(세액공제용 아님 유지)
완료 기준RECEIVED 물품을 타우리에서 영수증 발급 → 웹 후원자 화면에 증빙 묶음 노출.
규모중 — 도메인 전환 1 + API 1~2 + 화면 2.

5. 후원 취소 · 환불 정합성(구매 전 취소) 프로덕션 P0 ⬜ 미구현 BE WEB

선행"취소 허용 범위" 정책 확정 필요(기본선: 구매 BUYING 전까지 허용).
현재취소/환불 경로 없음. 상태값 RECORDED 하나뿐, 롤백·잠금 해제 규칙 미정.
할 일
  • BE: ContributionStatus.CANCELLED, 취소 허용 = BUYING 전, 비관락 raisedAmount 차감, 미달 시 LOCKED→RECRUITING 자동 복귀
  • API POST /api/contributions/{id}/cancel + 웹 후원내역 취소 버튼. 구매 후 환불은 별도(실결제 #3와 함께) — 범위 밖
완료 기준잠긴 물품에서 한 건 취소 → raisedAmount 감소 + 자동 RECRUITING 복귀 + 동시성 안전.
규모중.

6. 관리자 권한 세분화(permission authz) 프로덕션 P1 ⬜ 미구현 BE

왜 지금permission·역할-권한 도메인·관리 화면이 이미 있음(①) — 배선만 = ROI 높음. 방금 시드한 시설/플랫폼 관리자 계정으로 즉시 검증.
현재SecurityConfig는 hasAnyRole 역할 단위만. @PreAuthorize 거의 없음.
할 일권한 코드(PURCHASE_EXECUTE·FACILITY_WRITE 등) 시더 확장 → 민감 엔드포인트 @PreAuthorize("hasAuthority(...)") → JWT 권한 클레임.
완료 기준시설 관리자로 통합구매 실행 API → 403, 플랫폼 관리자 → 200.
규모중 — 도메인 존재, 배선 위주.

7. 상태 전환 감사 로그 · 이력 운영 P1 ⬜ 미구현 BE APP

왜 지금"감사메시지 = 프로덕션 필수"(§3). #6 authz와 운영 위생 세트.
현재물품 상태 전환(LOCKED→BUYING→SHIPPING→RECEIVED)에 누가·언제·왜 이력 없음 — updatedAt만.
할 일BE: StatusHistory(대상·이전→이후·actor·사유·시각) 추가 → 전환 시 append. APP: 타우리 타임라인 조회.
완료 기준물품 상세에서 상태 전환 이력이 타임라인으로 보인다.
규모소~중.

8. 마감일(deadline) 만료 · 목표미달 처리 운영 P2 ⬜ 미구현 BE

현재DonationItem.deadline 필드만. 만료 자동처리·미달 마감 규칙 없음 — 지난 물품도 계속 "모집중".
할 일경과 스케줄러/조회 파생상태 → 미달이면 EXPIRED + 후원분 환불과 연동.
의존취소·환불(#5) 롤백이 선행돼야 만료 환불 성립.
규모중.

9. 실구매액 ↔ 목표액 차액 정산 운영 P2 ⬜ 미구현 BE WEB

현재purchasedAmount를 목표액과 다르게 기록 가능하나, 잉여/부족 차액 처리·표시 미정.
할 일차액 정책(잔액 = 시설 후원금 귀속 등) + 구매현황에 "목표 vs 실구매 · 잔액" 표기.
의존실결제(#3) 붙으면 즉시 실이슈화 — 관문 이후.
규모중.

10. 인증 · 인가 견고화(401/403) 운영 P2 ⬜ 미구현 BE

왜 지금#6 authz의 하부 위생. 작아서 끼워넣기 좋음.
현재권한 거부 시 403 대신 401 등 상태코드 미세 결함. 리프레시 실패·연속실패 잠금 UX 얕음.
할 일인증 실패 = 401 / 권한 부족 = 403 정합화, 리프레시·잠금 정비.
규모소.
