Step 999 이행 루프 본체 6/6 미비한 필수 10

아직 미비한 필수 기능 · 베스트 10

step5(통합구매) + 웹 물품 구매현황 + step6(배송 딥링크)까지 구현되어 후원 이행 루프의 본체 6개는 동작한다. 이 문서는 그 위에서 실서비스 전 반드시 채워야 할 필수 기능만 기획 문서 근거와 함께 우선순위로 정리한다. 커뮤니티·명예관·통계·i18n 같은 순수 부가는 제외한다.

0. 지금 이행 루프는 어디까지 닫혔나

근거: 제품 핵심 기능 및 범위.html §5·§7·§8 이행 루프

시설 준비물 목표→ 공동충당→ 목표달성 자동잠금→ 통합구매 1회→ 송장·배송조회→ 수령확인→ 영수증 발급 ✗
루프의 마지막 칸(영수증 발급 · RECEIPTED)이 비어 있다. 상태값은 enum에 있으나 DonationItem.markReceipted() 전환 메서드도, 발급/증빙 도메인도, 후원자 노출도 없다. 본체를 진짜로 닫으려면 이 한 칸(아래 #1)이 먼저다.
본체 핵심 루프 완결에 필요 실서비스 실운영 전 반드시(부가지만 필수) 신뢰·운영 안정 운영을 위한 보강

1. 베스트 10 요약

#기능분류우선현재 상태(한 줄)
1영수증 발급 단계(RECEIPTED)본체P0enum만 존재, 전환·발급·노출 전무 — 루프 미완결

4증빙/영수증 파일 첨부 + 노출본체P1S3·필드 준비됨, 업로드 UI·화면 노출 미연결

7후원자 알림(단계 이벤트 통지)신뢰·운영P1후원자는 pull(재방문)만, push 통지 없음

P0(본체·실서비스 핵심): #1 루프 완결 → #2·#3 실돈 흐름. P1: 운영 신뢰의 뼈대. P2: 있으면 좋은 견고화. 전문가 검토 관문(§5·F-016): 실결제·환불(#2·#3)은 코드 이전에 회계/법률 검토가 선행 조건.

2. 상세 — 근거 · 현재 · 필요 이유 · 제안

1 영수증 발급 단계(RECEIPTED) 완성 본체 P0

기획 근거
이행 루프 마지막 칸 "영수증 발급자료 → 후원자 화면 단계 반영"(§5·§8). 정책 "영수증·구매 이행 추적은 단계적 확장".
현재
ItemStatus.RECEIPTED enum·메뉴(ADMIN_RECEIPTS)는 있으나 전환 메서드/도메인/후원자 노출 전무. RECEIVED에서 멈춘다.
왜 필수
이 칸이 비면 본체 루프가 열린 채다. 후원자가 "내 후원이 실제 물품으로 전달·정산됐다"를 확인하는 종착점.
제안
markReceipted() + 수령확인 후 증빙 묶음(구매영수증·배송완료·수령확인)을 발급자료로 정리 → 웹 구매현황/후원내역 타임라인에 "영수증" 노출. 세액공제용 아님 문구 유지(F-015).

2 실결제(PG) 연동 실서비스 P0

기획 근거
정책 "결제 기능은 전문가 검토 후 별도 단계에서 도입". ContributionStatus 주석: "차기 PAID 실결제 — 관문 통과 후".
현재
공동충당은 RECORDED(가상 리허설) — 실제 돈 이동 없음. 목표달성도 "기록상" 달성.
왜 필수
돈이 실제로 모이지 않으면 통합구매 재원이 없다. 실서비스의 전제.
제안
PG 선정(정기결제 불필요, 단건) → PAID 상태 + 결제키/승인번호 저장. 후원금은 시설 전용계좌 직접 귀속 구조(정책) 반영. 관문(F-016) 선행.

3 후원 취소 · 환불 정합성 실서비스 P0

기획 근거
ContributionStatus 주석: "차기 REFUNDED 환불". §5 정합성(초과·동시성)의 역방향 짝.
현재
취소/환불 경로 없음. 상태값은 RECORDED 하나뿐, raisedAmount 롤백·목표달성 후 환불 규칙 미정.
왜 필수
실결제(#2)가 붙는 순간 환불 요청은 반드시 발생. 잠금(LOCKED) 이후 환불 시 목표 재개방/구매 취소 규칙이 없으면 정합성이 깨진다.
제안
취소는 구매 실행(BUYING) 전까지만 허용을 기본선으로. 취소 시 비관락으로 raisedAmount 차감 + 목표 미달 시 RECRUITING 복귀. 구매 후 환불은 별도 정책.

4 증빙/영수증 파일 첨부 + 화면 노출 본체 P1

기획 근거
핵심기능표 "물품 구매 현황 = 구매 진행·완료, 영수증·송장 표시". 상태 전환의 신뢰 자료.
현재
evidenceFileUrl 필드 + UploadController/S3 인프라 준비됨. 그러나 통합구매 드로워에 업로드 UI 미연결, 구매현황 카드에 파일 노출 없음.
왜 필수
주문번호 텍스트만으론 신뢰가 약하다. 실제 영수증 이미지/PDF가 후원자에게 보여야 루프가 "증빙"으로 닫힌다.
제안
통합구매/송장 드로워에 파일 첨부(기존 shared/api/upload 재사용) → 구매현황 PurchaseStatusCard에 증빙 파일 링크 추가.

5 관리자 권한 세분화(permission 기반 authz) 실서비스 P1

기획 근거
범위 문서 "관리자 authz 세분화 = 프로덕션 필수 부가". permission·role-permission 도메인이 이미 설계됨.
현재
SecurityConfig는 hasAnyRole("PLATFORM_ADMIN","ADMIN") 역할 단위만(10곳). Permission/역할-권한 매핑 도메인은 있으나 authz에 미활용, @PreAuthorize 0건.
왜 필수
운영자가 늘면 "시설 등록만", "구매 실행만" 같은 세분 권한이 필요. 지금은 all-or-nothing.
제안
권한 코드(예 PURCHASE_EXECUTE, FACILITY_WRITE) 정의 → @PreAuthorize("hasAuthority(...)")로 전환. 기존 role-permission 매핑 화면 재활용.

6 상태 전환 감사 로그 · 이력 신뢰·운영 P1

기획 근거
범위 문서 "감사메시지 = 프로덕션 필수 부가". 정책 "신뢰·운영 가능성 우선".
현재
물품 상태 전환(LOCKED→BUYING→SHIPPING→RECEIVED)에 누가·언제·왜 이력이 없다. updatedAt 한 칸뿐.
왜 필수
돈·물품이 오가는 이행 루프에서 추적 불가능한 상태 변경은 분쟁 시 치명적. 오작동·오조작 역추적 근거.
제안
StatusHistory(대상·이전→이후·actor·사유·시각) 도메인 추가. 전환 메서드에서 append. Tauri에 타임라인 조회.

7 후원자 알림(단계 이벤트 통지) 신뢰·운영 P1

기획 근거
루프 종단 "후원자 화면에 단계 반영"(§8). 현재는 반영 방식이 재방문(pull)뿐.
현재
목표달성·구매·배송·수령 시 후원자에게 알리는 채널 없음. 후원자가 직접 들어와 확인해야 함.
왜 필수
내 후원이 어떻게 됐는지 모르면 재후원·신뢰가 안 쌓인다. 이행 루프의 "닫힘"을 후원자가 체감하는 장치.
제안
MVP는 인앱 알림함(이벤트 저장 + 뱃지)부터. 이메일/푸시는 이후. 단계 전환 훅에서 후원자별 알림 생성.

8 마감일(deadline) 만료 · 목표미달 처리 신뢰·운영 P2

기획 근거
물품 정책 "물품명…목표금액, 마감일을 관리한다".
현재
DonationItem.deadline 필드만 존재. 만료 시 자동 처리·목표 미달 마감 규칙 없음. 지난 물품도 계속 모집중.
왜 필수
마감이 지나도 계속 후원받으면 후원자 기대와 어긋난다. 미달 마감분의 환불(#3)과도 연동.
제안
마감 경과 스케줄러 or 조회 시 파생 상태 → 미달이면 EXPIRED(모집종료) 처리 + 후원분 환불 정책 연결.

9 실구매액 ↔ 목표액 차액 정산 규칙 신뢰·운영 P2

기획 근거
물품 정책 "목표금액은 실제 구매 총액 기준으로 표시".
현재
통합구매는 purchasedAmount(실구매액)를 목표액과 다르게 기록 가능하나, 잉여/부족 차액 처리 규칙·표시가 미정.
왜 필수
모인 돈 > 실구매액이면 잔액 귀속, 반대면 부족분 처리 — 회계 투명성의 핵심. 실결제(#2) 붙으면 즉시 이슈화.
제안
차액 정책 정의(잔액=시설 후원금 귀속 등) + 구매현황에 "목표 vs 실구매" 비교·잔액 표기.

10 인증 · 인가 견고화 신뢰·운영 P2

기획 근거
범위 문서 "관리자 authz 세분화·감사 = 실서비스 필수". #5의 하부 위생.
현재
권한 거부 시 403이 아닌 401로 응답되는 등 상태코드 미세 결함(기능 차단은 정상). 토큰 리프레시는 있으나 실패·만료 UX·계정 잠금 등은 얕음.
왜 필수
실서비스에서 인증 실패 응답이 부정확하면 클라이언트 오류 처리·모니터링이 꼬인다.
제안
인증 실패=401 / 권한 부족=403 정합화, 리프레시 실패 흐름·연속 실패 잠금 정비.
