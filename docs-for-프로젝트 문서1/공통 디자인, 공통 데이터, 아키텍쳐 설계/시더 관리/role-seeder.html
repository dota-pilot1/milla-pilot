DonationPlatform Role Seeder Light Mode Only

역할 시더 관련 로직 문서

이 문서는 DonationPlatform 서버 시작 시 기본 역할을 자동으로 준비하는 시더 로직과, “DB에 없을 경우에만 다시 추가”하는 방식이 어떻게 동작하는지 정리한다.

결론

역할 데이터는 직접 DB에 수동 입력하기보다 RoleSeeder에서 관리하는 것이 합리적이다. 서버가 시작될 때 기준 역할을 확인하고, 존재하지 않는 역할만 추가하면 로컬 개발 DB, 테스트 DB, 신규 환경에서 역할 누락 문제를 줄일 수 있다.

현재 기준 역할은 ROLE_DONOR, ROLE_FACILITY_ADMIN, ROLE_PLATFORM_ADMIN 세 가지다.

역할 정의

Donor ROLE_DONOR
웹에서 후원과 후원 내역을 이용하는 사용자.
Facility Admin ROLE_FACILITY_ADMIN
시설 기준 물품 요청과 수령 업무를 처리하는 사용자.
Platform Admin ROLE_PLATFORM_ADMIN
플랫폼 전체 운영과 시스템 관리를 담당하는 사용자.

관련 파일

파일 역할
donation-platform-server/src/main/java/com/cj/donationplatform/config/RoleSeeder.java 서버 시작 시 기본 역할을 확인하고 없으면 생성한다.
donation-platform-server/src/main/java/com/cj/donationplatform/user/application/AuthService.java 회원가입 시 첫 계정은 플랫폼 관리자, 이후 계정은 후원자로 지정한다.
donation-platform-server/src/main/java/com/cj/donationplatform/config/MenuSeeder.java 관리자 메뉴의 접근 역할을 ROLE_PLATFORM_ADMIN 기준으로 연결한다.

시더 실행 방식

RoleSeeder는 Spring Boot의 ApplicationRunner를 구현한다. 따라서 애플리케이션 컨텍스트가 준비된 뒤 서버 시작 과정에서 자동 실행된다. 

@Component
@Order(2)
@RequiredArgsConstructor
public class RoleSeeder implements ApplicationRunner {
    @Override
    @Transactional
    public void run(ApplicationArguments args) {
        // 기본 역할 확인 및 생성
    }
}
@Order(2)는 여러 시더가 있을 때 실행 순서를 지정한다. 역할은 메뉴나 권한 매핑의 기준 데이터이므로 메뉴 시더보다 먼저 실행되는 것이 자연스럽다.

없을 경우에만 추가하는 로직

핵심은 roleRepository.existsByCode(def.code()) 검사다. 역할 코드는 고유값이므로, 같은 코드가 이미 DB에 있으면 새로 저장하지 않는다. 

for (RoleDef def : defaults) {
    if (!roleRepository.existsByCode(def.code())) {
        roleRepository.save(
            Role.create(def.code(), def.name(), def.description(), true)
        );
        log.info("Seeded role: {} ({})", def.code(), def.name());
    }
}
  1. 기본 역할 목록을 defaults에 정의한다.
  2. 각 역할의 code가 DB에 있는지 확인한다.
  3. 없으면 Role.create(...)로 새 역할 엔티티를 만든다.
  4. 이미 있으면 저장하지 않고 그대로 둔다.
이 방식은 idempotent 하다. 즉 서버를 여러 번 재시작해도 같은 역할이 중복 생성되지 않는다.

현재 역할 시더 기준값

List<RoleDef> defaults = List.of(
    new RoleDef(ROLE_DONOR, "후원자", "웹에서 후원과 후원 내역을 이용하는 사용자"),
    new RoleDef(ROLE_FACILITY_ADMIN, "시설 관리자", "시설 기준 물품 요청과 수령 업무를 처리하는 사용자"),
    new RoleDef(ROLE_PLATFORM_ADMIN, "플랫폼 관리자", "DonationPlatform 전체 운영과 시스템 관리를 담당하는 사용자")
);

새 역할 코드는 도메인 의미가 명확하다. 기존의 ROLE_USER, ROLE_MANAGER, ROLE_ADMIN보다 후원 플랫폼의 실제 업무 역할을 더 직접적으로 표현한다.

회원가입 기본 역할

AuthService.signup에서는 가입 시점의 사용자 수를 확인한다. 사용자가 한 명도 없다면 첫 계정은 시스템 초기 운영을 위해 ROLE_PLATFORM_ADMIN이 된다. 이후 가입자는 기본적으로 ROLE_DONOR가 된다. 

String signupRoleCode = userRepository.count() == 0
        ? RoleSeeder.ROLE_PLATFORM_ADMIN
        : RoleSeeder.ROLE_DONOR;
시설 관리자는 일반 회원가입으로 자동 배정하지 않는다. 시설 인증, 시설 등록, 운영자 승인 흐름이 필요하므로 플랫폼 관리자가 사용자 관리 화면에서 역할을 변경하는 방식이 더 안전하다.

메뉴 시더와의 관계

관리자 메뉴는 MenuSeeder에서 RoleSeeder.ROLE_PLATFORM_ADMIN을 requiredRole로 사용한다. 이렇게 하면 데스크탑 관리자 앱은 플랫폼 관리자에게 필요한 메뉴만 노출할 수 있다. 

new MenuDef(
    "ADMIN_USERS",
    "ADMIN_SYSTEM",
    "유저 관리",
    "nav.users",
    "/users",
    "Users",
    RoleSeeder.ROLE_PLATFORM_ADMIN,
    0
)

기존 역할 호환 정책

현재 로컬 개발 데이터에는 기존 ROLE_ADMIN 계정이 남아 있을 수 있다. 그래서 관리자 API는 새 역할인 ROLE_PLATFORM_ADMIN과 기존 역할인 ROLE_ADMIN을 함께 허용한다. 

@PreAuthorize("hasAnyRole('ADMIN', 'PLATFORM_ADMIN')")

이 호환 처리는 기존 개발 계정을 즉시 깨뜨리지 않기 위한 장치다. 운영 데이터 마이그레이션이 끝나면 ROLE_ADMIN 허용을 제거하고 ROLE_PLATFORM_ADMIN만 남기는 방향으로 정리할 수 있다.

운영 시 주의점

  • 시더는 기본 데이터를 보장하기 위한 장치이며, 운영 중인 역할 이름을 무조건 덮어쓰지 않는다.
  • 현재 로직은 역할이 없을 때만 생성한다. 이미 존재하는 역할의 이름이나 설명은 자동 수정하지 않는다.
  • 역할 코드 변경은 사용자, 메뉴, 권한 매핑, JWT 인증 흐름에 영향을 주므로 신중하게 처리해야 한다.
  • 시설 관리자 자동 배정은 아직 하지 않는 것이 낫다. 시설 소유 검증이나 승인 프로세스가 필요하다.
