DonationPlatform Architecture

백엔드 아키텍쳐 설계

Spring Boot 기반 API 서버의 현재 구조, 도메인 경계, 인증·권한 흐름, 데이터 정책, 그리고 MVP 이후 보완해야 할 항목을 정리한다.

RuntimeJava 21 / Spring Boot 4.0.5
DataPostgreSQL / Spring Data JPA
SecurityJWT Access + Refresh Token
API Port4301

1. 아키텍처 개요

Controller
DTO
Service
Domain Entity
Repository

현재 백엔드는 도메인별 패키지 안에서 presentation, application, domain, infrastructure 레이어를 나누는 레이어드 모듈 구조다. 완전한 Clean Architecture라기보다 Spring Boot/JPA 생산성을 우선한 도메인 중심 레이어드 구조로 보는 것이 정확하다.

├── facility
├── donation_item
└── contribution

2. 도메인 모듈

모듈 책임 주요 파일
auth JWT 생성·검증, RefreshToken 저장, 인증 필터, Spring Security 사용자 어댑터 JwtTokenProvider, JwtAuthenticationFilter, RefreshToken
user 회원가입, 로그인, 유저 관리, 프로필, 역할 연결 AuthService, UserManagementService, User
role, permission 역할/권한/권한 카테고리 CRUD 및 운영자 권한 모델 RoleSeeder, PermissionSeeder, RolePermissionManager 연동 API
menu DB 기반 웹/관리자 메뉴 구성, 권한별 메뉴 노출 MenuSeeder, MenuController, MenuService
facility 후원 대상 시설 CRUD, 공개 시설 목록, 관리자 시설 순서 관리 Facility, FacilityService, FacilityController
donation_item 시설별 후원 물품 CRUD, 목표금액/진행상태/카테고리 관리 DonationItem, DonationItemService, ItemStatus
contribution 후원 참여, 내 후원 내역, 물품 목표금액 반영 Contribution, ContributionService, ContributionController
site_settings 로그인/회원가입 화면 등 사이트 공통 설정 SiteSetting, SiteSettingController
common 공통 예외, 공통 응답, S3 업로드 GlobalExceptionHandler, ErrorCode, UploadService

3. 레이어 규칙

presentation

HTTP endpoint, request validation, response DTO 변환을 담당한다. 비즈니스 판단은 넣지 않는다.

application

트랜잭션 경계와 유스케이스 흐름을 담당한다. 여러 Repository 호출과 도메인 상태 변경을 조율한다.

domain

JPA Entity, enum, 상태 변경 메서드 등 도메인 규칙을 둔다. 외부 API 호출은 넣지 않는다.

infrastructure

Spring Data Repository와 외부 저장소 연동을 둔다. 현재는 JPA Repository 중심이다.

4. 인증·권한 흐름

로그인 요청
AuthService
JWT 발급
RefreshToken 저장
Bearer 인증
  • /api/auth/signup, /api/auth/login, /api/auth/refresh, /api/auth/check-email은 공개 API다.
  • 공개 조회 API: GET /api/site-settings, GET /api/menus, GET /api/facilities/**, GET /api/donation-items/**.
  • 그 외 API는 JWT 인증이 필요하다.
  • 관리자 기능은 @PreAuthorize와 Role 기반 인가로 보호한다.
  • CORS 허용 origin은 http://localhost:4300, http://localhost:1422, tauri://localhost이다.

5. 데이터 모델 핵심

Aggregate 핵심 속성 현재 정책
User email, passwordHash, username, role, active 역할 기반 화면/API 접근. 후원자 시더는 donor01@test.com ~ donor10@test.com.
Facility code, name, region, type, status, verified, avatarInitial, displayOrder MVP에서는 시설 이미지를 노출하지 않고 카드 이니셜을 사용한다.
DonationItem facility, category, goalAmount, raisedAmount, targetQuantity, status, deadline MVP에서는 이모지 썸네일을 사용하고, 목표금액 기준 진행률을 표시한다.
Contribution user, item, amount, status 내 후원 내역과 물품 목표 달성 흐름의 근거 데이터다.
Menu code, label, path, icon, displayOrder, role/permission 사이드바와 관리자 메뉴를 DB 기준으로 운영한다.

6. 시더 전략

권한/메뉴

RoleSeeder, PermissionSeeder, MenuSeeder가 기준 데이터를 생성한다.

후원 도메인

FacilitySeeder, DonorUserSeeder, ContributionSeeder가 데모 데이터를 만든다.

멱등성

대부분 code/email 기준으로 중복 생성을 방지한다. 운영 전 멱등 기준을 일관화해야 한다.

7. API 설계 원칙

  • CRUD API는 도메인별 Controller에 둔다.
  • Request DTO는 validation annotation으로 1차 검증한다.
  • Response DTO는 Entity 직접 노출을 피하고 정적 변환 메서드로 만든다.
  • 예외는 BusinessException, ErrorCode, GlobalExceptionHandler로 표준화한다.
  • 파일 업로드는 서버 직접 업로드보다 S3 presigned URL 발급 구조를 사용한다.

8. 보완 필요한 부분

우선순위 높은 보완
  • 후원 목표금액 초과 방지: 동시 후원 요청에 대한 DB lock, optimistic/pessimistic locking, 트랜잭션 테스트가 필요하다.
  • DonationItem.raisedAmount는 Contribution 합계와 정합성이 깨질 수 있으므로 업데이트 정책을 명확히 해야 한다.
  • 상태 전이 규칙: RECRUITING → LOCKED → BUYING → SHIPPING → RECEIVED → RECEIPTED 전이를 서버에서 강제해야 한다.
  • 관리자/시설관리자/후원자별 API 권한 매트릭스를 문서화하고 테스트해야 한다.

구조 보완

  • 현재 서비스 클래스가 CRUD와 정책 판단을 같이 들고 있다. 후원 결제/구매/배송이 커지면 도메인 서비스 또는 유스케이스 클래스로 분리한다.
  • 시더는 개발/데모용과 운영 기준 데이터를 구분하는 profile 전략이 필요하다.
  • JPA ddl-auto: update는 MVP 개발에는 편하지만 운영 전 Flyway/Liquibase 같은 migration 도구로 전환해야 한다.
  • RefreshToken 저장 정책에 재사용 감지, 로그아웃, 만료 정리 배치가 필요하다.
  • 공개 API와 관리자 API의 rate limit, audit log, 변경 이력 테이블을 검토한다.

테스트 보완

  • Controller 단위 API contract 테스트.
  • Service 트랜잭션/동시성 테스트.
  • SecurityConfig 인가 테스트.
  • Seeder 멱등성 테스트.

9. 유지보수 체크리스트

  • 새 도메인은 presentation/application/domain/infrastructure 구조를 따른다.
  • Entity를 Controller에서 직접 반환하지 않는다.
  • 상태 enum은 사용자 라벨과 분리한다. 라벨 변경은 프론트/문서에서 처리한다.
  • 시더에 테스트 계정이 추가되면 공통 관리/시더 관리 문서와 로그인 개발용 계정 UI를 같이 갱신한다.
  • 후원/구매/배송 정책은 기획 문서/후원 관련 정책 요약을 기준 문서로 삼는다.
