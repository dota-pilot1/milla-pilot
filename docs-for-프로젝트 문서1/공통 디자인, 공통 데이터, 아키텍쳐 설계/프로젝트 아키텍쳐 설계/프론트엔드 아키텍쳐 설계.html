DonationPlatform Architecture

프론트엔드 아키텍쳐 설계

DonationPlatform은 후원자 웹 프론트와 Tauri 관리자 앱을 분리해 운영한다. 두 앱은 같은 백엔드 API를 바라보지만, 화면 목적과 UX 밀도가 다르므로 FSD 계층과 공통 UI 규칙을 기준으로 관리한다.

WebNext.js 16 / React 19 / App Router
AdminTauri 2 / Vite 7 / React 19
StateTanStack Query / Store / React Hook Form
DesignTailwind CSS 4 / Radix / lucide

1. 앱 구성

후원자 웹 프론트

경로: donation-platform-front
역할: 후원하기, 시설/물품 상세, 내 후원 내역, 로그인/회원가입, 관리자 웹 CRUD 화면 일부.

Tauri 관리자 앱

경로: donation-admin-tauri
역할: 데스크톱 운영 콘솔, 시설/후원 물품 관리, DB 메뉴 기반 사이드바, 앱 업데이트 패널.

웹 프론트는 일반 사용자와 웹 관리자 화면을 포함하고, Tauri 앱은 운영자 반복 업무에 맞춘 밀도 높은 CRUD 화면을 담당한다. 둘 다 백엔드 http://localhost:4301 API와 통신한다.

2. 웹 프론트 계층 구조

웹 프론트는 Feature-Sliced Design에 가까운 계층을 사용한다. 현재 구현은 app, widgets, features, entities, shared로 나뉜다. 

donation-platform-front/src
├── app                 Next App Router 페이지와 앱 Provider
├── widgets             Header, Guard 등 페이지 조립 단위
├── features            사용자 액션 단위 UI/상태
├── entities            도메인 API, 타입, 도메인 카드 UI
└── shared              공통 API client, lib, validation, UI primitive
계층 책임 예시
app 라우팅, 페이지 composition, 전역 Provider, 인증 초기화 /donate, /my-donations, layout.tsx
widgets 여러 feature/entity를 묶는 화면 프레임 Header, RequireAuth, RequireRole
features 로그인, 후원하기, 필터링, 내 후원 내역 등 사용자 행위 단위 DonatePanel, FacilityFilterBar, MyDonationsList
entities 도메인 모델 타입, API, 단순 도메인 UI facility, donation-item, contribution, user
shared 도메인 독립 공통 코드 Button, Dialog, Badge, api, formatKRW

3. Tauri 관리자 앱 계층 구조

Tauri 관리자 앱도 웹과 유사하게 app, widgets, features, entities, shared 구조를 따른다. 다만 App Router가 없고, App.tsx가 메뉴 상태를 기준으로 화면을 분기한다. 

donation-admin-tauri/src
├── app                 앱 부트스트랩, 메뉴 모델, 화면 분기
├── widgets/app-shell   사이드바, 탑바, 윈도우 컨트롤
├── features            시설 관리, 후원 물품 관리, 메뉴 API, 로그인
├── entities            facility, donation-item, user 타입
└── shared              API client, config, 공통 UI primitive
Tauri window는 macOS 화면 분할 기능을 위해 decorations: true, titleBarStyle: Overlay, hiddenTitle: true, trafficLightPosition 숨김 방식을 사용한다.

4. 데이터 흐름

Page
Feature UI
Entity API
shared API Client
Spring API

웹 프론트

  • 서버 상태는 @tanstack/react-query로 조회/캐싱한다.
  • API 통신은 shared/api/axios.ts의 axios instance를 사용한다.
  • JWT access token은 request interceptor에서 자동 첨부한다.
  • 401 발생 시 refresh token으로 재발급하고 pending request queue를 flush한다.
  • 인증 상태는 entities/user/model/authStore와 AuthInitializer가 관리한다.

Tauri 관리자

  • shared/api/client.ts의 apiRequest를 통해 API 호출을 통일한다.
  • 관리자 JWT는 각 feature API 호출에 token으로 전달한다.
  • 서버 주소는 shared/config/server에서 관리한다.

5. 라우팅과 화면 책임

영역 대표 경로 책임
인증 /login, /register 로그인, 회원가입, 개발용 시더 계정 빠른 입력
후원하기 /donate, /donate/facilities/[id], /donate/items/[id] 시설 탐색, 물품 진행률 확인, 후원 참여
내 후원 내역 /my-donations 시설별 내 후원 목록, 진행 단계 타임라인 확인
관리자 웹 /users, /roles, /permissions, /menu-management 유저/권한/메뉴/사이트 설정 관리
Tauri 관리자 DB 메뉴 기반 화면 분기 운영자용 시설·후원 물품 CRUD와 순서 관리

6. 공통 UI 시스템

디자인 시스템은 라이브러리 전환이 아니라 “AI와 개발자가 읽는 운영 규칙”으로 둔다. 기준 문서는 docs/design-system.md와 docs/frontend-conventions.md이다.

Primitive

Button, Card, Dialog, Badge, Progress, FormField 등.

Status UI

상태 색은 Badge variant를 단일 출처로 사용한다.

Icon

버튼/메뉴/툴은 가능한 lucide-react 아이콘을 사용한다.

현재 상태 라벨

  • 시설/물품 진행: 모집중, 통합 구매 진행, 배송중, 목표달성
  • 후원 이행 타임라인: 모집중 → 목표달성 → 통합 구매 진행 → 배송중 → 수령확인 → 영수증
  • 시설 이미지는 MVP에서 노출하지 않고 이니셜을 사용한다.
  • 후원 물품은 MVP에서 이모지 썸네일을 사용한다.

7. 상태 관리 전략

상태 종류 도구 사용 기준

8. 보완 필요한 부분

우선순위 높은 보완
  • next.config.ts가 production에서 output: export를 사용한다. 동적 라우트 /donate/items/[id] 등은 정적 export와 충돌하므로 배포 전략을 결정해야 한다.
  • 웹 프론트와 Tauri 관리자 앱의 공통 타입/API 스키마가 수동 중복이다. OpenAPI 기반 타입 생성 또는 공유 패키지 검토가 필요하다.
  • 후원 상태 enum 라벨이 일부 화면에서 중복될 수 있다. 상태 라벨/색상 매핑 단일 출처를 강화해야 한다.
  • 로그인 개발용 시더 계정 UI는 운영 빌드에서 숨기지만, 환경별 노출 조건을 문서화하고 테스트해야 한다.

FSD 구조 보완

  • features/my-donations/ui/MyDonationsList.tsx는 아코디언/row/timeline 조립이 커지고 있다. DonationFacilityAccordion, DonationContributionRow, shared/ui/Collapsible로 분리하는 것이 좋다.
  • 애니메이션 접힘/펼침 로직은 도메인과 무관하므로 feature 내부보다 shared/ui/Collapsible가 적절하다.
  • 관리자 웹 CRUD와 Tauri CRUD의 UI 원칙을 문서상 같은 기준으로 묶되, 코드 공유는 서두르지 않는다.
  • page 파일은 data loading과 composition에 집중하고, 복잡한 렌더링은 feature로 내린다.

UX/성능 보완

  • 아코디언은 내부 타임라인이 많아 height animation이 흔들릴 수 있다. 공통 Collapsible에서 measured height, unmount, motion-reduce 정책을 관리한다.
  • 후원 물품 카드의 진행바/금액/버튼 정렬 같은 고정 포맷 UI는 stable grid를 사용한다.
  • 리스트가 길어지면 pagination 또는 virtualization을 검토한다.
  • 이미지 도입 시 시설 이미지는 개인정보/보호 정책 확정 후 적용하고, 물품 이미지부터 우선 검토한다.

테스트 보완

  • 로그인/토큰 refresh interceptor 테스트.
  • 후원하기 → 결제/후원 → 내 후원 내역 E2E 테스트.
  • 권한별 메뉴 노출 테스트.
  • Tauri 시설 순서 drag-and-drop 회귀 테스트.

9. 구현 규칙

  • 새 화면은 app에 page를 만들고, 실제 화면 로직은 features로 분리한다.
  • 도메인 타입과 API 호출은 entities/{domain}에 둔다.
  • 공통 primitive가 없으면 shared/ui에 먼저 만든다.
  • 상태 색상은 페이지에서 직접 Tailwind palette를 쓰지 말고 공통 variant를 사용한다.
  • 페이지/feature에서 raw input/select/table 스타일을 반복하지 않는다.
  • 프론트와 Tauri에서 같은 정책을 보여줘야 하는 라벨은 문서와 타입 상수를 함께 갱신한다.
  • 사용자 화면은 설명을 과하게 늘리지 않고, 필요할 때 안내 다이얼로그나 help affordance로 보완한다.
