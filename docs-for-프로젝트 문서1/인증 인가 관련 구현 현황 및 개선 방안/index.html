인증·인가 구현 현황 및 개선 방안

DonationPlatform의 인증·인가 체계는 JWT 기반 stateless 인증과 Role 기반 API 보호를 중심으로 구현되어 있다. Refresh Token 저장·회전, 메뉴 표시 권한, Permission 데이터 모델도 존재한다. 다만 현재 API 접근 제어는 아직 Permission 중심까지 완전히 내려오지 않았고, 운영 보안 관점에서는 토큰 저장 방식·감사 로그·시설 관리자 범위 제한을 보강해야 한다.

요약 판단

현재 MVP 적합성 플랫폼 관리자 중심의 내부 운영 MVP라면 현 구조로 동작 가능하다. 로그인, 토큰 발급, refresh, role 기반 보호가 갖춰져 있다.
가장 큰 미비점 Permission은 모델과 시더가 있지만, 실제 주요 API 보호는 대부분 ROLE_PLATFORM_ADMIN 중심이다.
우선 개선 방향 Refresh 재발급 안정화, 권한 매핑의 실제 적용, 시설 관리자 scope 제한, 인증 실패 UX 통일 순서가 합리적이다.
최근 관리자 앱은 access token 만료 시 /api/auth/refresh로 재발급 후 원 요청을 1회 재시도하도록 보강되었다. refresh token도 만료되었거나 서버 저장 토큰과 불일치하면 세션을 정리하고 로그인 화면으로 돌리는 구조다.

구현 현황 매트릭스

영역 상태 현재 구현 판단
회원가입/로그인 구현됨 /api/auth/signup, /api/auth/login. 첫 가입자는 플랫폼 관리자, 이후 기본 후원자 역할 부여. MVP에는 충분하다. 운영 전에는 관리자 최초 계정 생성 절차를 별도 통제하는 편이 좋다.
Access Token 구현됨 JWT claim에 user id, email, username, role, permissions, type 포함. 기본 만료는 설정값 기준 30분. stateless API 인증에 적합하다. 단, 권한 변경 즉시 반영은 refresh/재로그인 이후가 기준이다.
Refresh Token 구현됨 DB 저장형 refresh token. 재발급 시 새 refresh token으로 회전하고 기존 값을 대체한다. 단일 세션 기준으로 안전한 편이다. 다중 기기 세션을 지원하려면 userId 단일 저장 구조를 확장해야 한다.
JWT 필터 구현됨 Authorization: Bearer 토큰을 파싱하고 access token일 때만 SecurityContext를 구성한다. refresh token으로 API 접근이 되지 않도록 type을 구분하는 점은 적절하다.
API 인가 부분 구현 SecurityConfig와 일부 컨트롤러에서 hasAnyRole('PLATFORM_ADMIN', 'ADMIN') 중심으로 보호한다. 관리자 MVP는 가능하지만, 세부 권한 관리 화면과 실제 API 권한 적용 사이에 간극이 있다.
Role/Permission 모델 부분 구현 roles, permissions, role_permissions 모델과 관리 API가 있다. 데이터 모델은 준비되어 있으나 실제 보호 규칙을 hasAuthority로 전환하는 작업이 남아 있다.
메뉴 권한 부분 구현 menus.requiredRole, requiredPermission 필드가 있고 앱에서 메뉴 접근 가능 여부를 계산한다. UI 노출 제어에는 유용하지만, 보안 경계는 반드시 서버 API 인가가 담당해야 한다.
웹 클라이언트 refresh 구현됨 Axios interceptor가 401 발생 시 refresh 후 원 요청을 재시도한다. 동시 refresh는 queue로 묶는다. 교과서적인 방식이다. refresh 실패 시 전역 logout 이벤트를 발생시키는 점도 적절하다.
Tauri 관리자 refresh 보강됨 apiRequest에서 401 발생 시 refresh token으로 재발급하고 원 요청을 1회 재시도한다. 최근 발생한 “유효하지 않은 토큰” 화면 문제의 핵심 원인을 해소하는 방향이다.

현재 인증 흐름

1. 로그인 클라이언트가 이메일/비밀번호를 /api/auth/login으로 전송한다.
2. 토큰 발급 서버가 access token과 refresh token을 발급하고 refresh token을 DB에 저장한다.
3. API 호출 보호 API는 Authorization: Bearer access token으로 호출한다.
4. 만료 감지 access token 만료로 401이 발생하면 클라이언트가 refresh API를 호출한다.
5. 재시도/로그아웃 재발급 성공 시 원 요청을 재시도하고, 실패 시 로컬 세션을 정리한다.
서버는 refresh token을 단순 검증만 하지 않고 DB에 저장된 값과 비교한다. 따라서 로그아웃, 토큰 회전, 서버측 세션 무효화가 가능하다.

인가 구조

구분 현재 방식 장점 한계
Role 기반 보호 hasAnyRole('PLATFORM_ADMIN', 'ADMIN') 단순하고 명확하다. 현재 관리자 콘솔 중심 구조에 잘 맞는다. 유저 관리, 롤 관리, 시설 관리처럼 세부 권한 분리가 필요한 순간 유연성이 떨어진다.
Permission 모델 USER_VIEW, ROLE_EDIT 등 권한 코드와 카테고리 시더 권한 관리 UI와 역할-권한 매핑의 기반이 있다. 실제 API annotation/URL rule에는 아직 제한적으로만 반영되어 있다.
메뉴 권한 메뉴별 requiredRole/requiredPermission 관리자 사이드바와 웹 메뉴 노출을 서버 데이터로 제어할 수 있다. 메뉴 숨김은 UX 제어일 뿐이다. 직접 URL/API 호출은 서버 인가가 막아야 한다.
시설 관리자 범위 ROLE_FACILITY_ADMIN 역할은 존재하나 API scope 제한은 미완성 도메인 역할은 이미 정의되어 있다. 시설별 담당자-시설 매핑과 row-level authorization이 필요하다.

주요 리스크

Permission과 실제 보호 규칙의 불일치 관리 화면에서 권한을 조정해도 주요 API가 role rule만 보면 운영자가 기대한 권한 분리가 적용되지 않을 수 있다.
Refresh token 저장 위치 현재 웹과 Tauri 모두 localStorage 계열 저장을 사용한다. MVP 개발 편의성은 좋지만 XSS·로컬 접근 리스크가 있다.
시설 단위 접근 제어 부재 시설 관리자가 도입되면 “내 시설의 물품/배송/증빙만 접근” 같은 row-level rule이 필수다.
권한 관리 UI가 존재하는 상태에서는 “권한을 설정했는데 실제 API가 왜 열려 있나?”라는 혼란이 생기기 쉽다. 단기적으로는 화면에 “현재 플랫폼 관리자 전용”임을 명확히 하고, 중기적으로는 Permission 기반 보호로 맞춰야 한다.

개선 우선순위

우선순위 작업 구체 방안 기대 효과
P0 토큰 만료 UX 안정화 모든 클라이언트 API 래퍼에서 401 refresh 재시도, 실패 시 세션 정리와 로그인 화면 이동을 일관 처리한다. 장시간 미사용 후 “유효하지 않은 토큰” 오류 화면에 멈추는 문제를 줄인다.
P0 관리 API 보호 범위 재점검 신규/기존 관리자 API가 authenticated()만 타지 않는지 확인하고 role/permission rule을 명시한다. 실수로 열린 운영 API를 방지한다.
P1 Permission 기반 전환 USER_VIEW, FACILITY_EDIT, CONTRIBUTION_VIEW 같은 도메인 권한을 정리하고 hasAuthority를 적용한다. 역할 이름이 아니라 실제 업무 권한 단위로 접근을 제어할 수 있다.
P1 시설 관리자 scope 설계 사용자-시설 매핑 테이블을 만들고 facility id 기준으로 조회/수정 범위를 제한한다. 시설 담당자에게 필요한 기능만 열 수 있다.
P2 감사 로그 로그인 실패, refresh 실패, 권한 변경, 관리자 데이터 수정 이벤트를 별도 감사 로그로 남긴다. 운영 사고 분석과 권한 오남용 추적이 쉬워진다.
P2 토큰 저장 정책 강화 웹은 httpOnly secure cookie 전환 가능성을 검토하고, Tauri는 OS secure storage 도입을 검토한다. 토큰 탈취 표면을 줄인다.

권장 목표 구조

  1. 로그인 성공 시 서버는 access token과 refresh token을 발급하고, refresh token은 서버 DB에 저장된 최신 값만 인정한다.
  2. 클라이언트는 access token 만료 시 refresh를 한 번 시도하고, 성공하면 원 요청을 한 번만 재시도한다.
  3. 서버 API는 URL rule과 method security 중 하나를 기준으로 권한 정책을 명확히 문서화한다.
  4. 관리 기능은 ROLE_PLATFORM_ADMIN 같은 큰 역할보다 FACILITY_EDIT, CONTRIBUTION_VIEW 같은 업무 권한으로 점진 전환한다.
  5. 시설 관리자 기능이 열릴 때는 role만 보지 말고 facility ownership/scope를 반드시 함께 검사한다.
