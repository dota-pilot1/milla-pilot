백엔드 개발 스펙 조사 및 적절성

DonationPlatform 백엔드는 후원 물품 공동구매 펀딩 시스템의 인증, 메뉴, 권한, 시설, 후원 물품, 업로드 API를 담당한다. 현재 구조는 단일 Spring Boot 애플리케이션이며, 1차 MVP 기준으로는 충분히 합리적이다.

기본 스펙

항목 현재 스펙 적절성 판단
프로젝트 donation-platform-server 프론트와 관리자 앱이 함께 쓰는 중앙 API 서버로 적절하다.
언어/런타임 Java 21 toolchain LTS 버전이라 장기 유지에 적합하다.
프레임워크 Spring Boot 4.0.5 최신 스택이다. 단, 라이브러리 호환성 이슈가 생기면 3.x LTS 계열 검토 여지는 있다.
DB PostgreSQL, host port 5436 결제/후원/정산성 데이터에는 RDB가 맞다.
ORM Spring Data JPA, Hibernate CRUD 중심 MVP에는 생산성이 높다. 복잡한 통계/정산 쿼리는 별도 query 전략이 필요하다.
인증 Spring Security, JWT access/refresh token 웹과 Tauri 앱이 같이 쓰기 좋다. refresh token 저장/폐기 정책은 운영 전에 강화해야 한다.
API 문서 springdoc-openapi 2.8.8 Swagger UI 기반 협업에 적합하다.
파일 업로드 AWS SDK S3, presigned URL 증빙/영수증 업로드까지 확장하기 좋다. 초기에는 환경 변수 누락에 주의한다.

주요 모듈 평가

모듈 역할 적절성 보강 포인트
auth, user 로그인, 회원가입, JWT, 사용자 관리 웹/Tauri 공통 인증 기반으로 적절하다. 역할별 접근 제어 테스트를 추가한다.
role, permission RBAC 관리 시설 관리자/플랫폼 관리자/후원자 구분에 필요하다. 역할 코드와 메뉴 권한 매핑 기준을 문서와 시더에 계속 맞춘다.
menu DB 기반 메뉴 관리 초기 운영형 프로젝트에는 유연하다. 웹 메뉴와 Tauri 메뉴를 테이블 또는 menuType으로 분리하는 설계가 필요하다.
facility 후원 대상 시설 관리 1차 MVP의 첫 핵심 도메인이다. 시설 공개 여부, 승인 상태, 담당자 매핑을 강화한다.
donation_item 시설별 필요 물품 관리 후원하기 페이지와 직접 연결되는 핵심 도메인이다. 목표 금액/수량, 달성률, 상태 전이를 엄격히 관리한다.
common/upload S3 presigned URL 발급 증빙 자료 업로드에 적합하다. 파일 확장자, 크기, 접근 권한 검증을 추가한다.

보안/인증 설정

  • Stateless JWT 인증을 사용한다.
  • 허용 origin은 http://localhost:4300, http://localhost:1422, tauri://localhost이다.
  • 공개 API는 회원가입, 로그인, 토큰 갱신, 이메일 확인, 사이트 설정 조회, 메뉴 조회, 시설 조회, Swagger다.
  • 그 외 API는 인증이 필요하다.
웹의 후원자 공개 화면에서 시설과 후원 물품을 조회해야 하므로 GET /api/facilities/** 공개는 자연스럽다. 단, 생성/수정/삭제는 관리자 권한으로 제한해야 한다.

DB/마이그레이션 평가

항목 현재 판단
DDL spring.jpa.hibernate.ddl-auto=update 개발 초기에는 편하지만 운영 전에는 Flyway 또는 Liquibase 도입이 필요하다.
시더 Role, Permission, Menu, Facility seeder 존재 초기 데이터 재현성이 좋아진다. idempotent upsert 유지가 중요하다.
테스트 DB H2 test runtime 단위 테스트에는 적절하지만 PostgreSQL 전용 동작은 Testcontainers 검토가 필요하다.

MVP 적합성 결론

현재 선택은 적절함 Spring Boot + JPA + PostgreSQL 조합은 후원/결제/상태 추적성 데이터에 맞다.
과하지 않음 도메인 규모가 MES나 쇼핑몰보다 작아 단일 모듈 백엔드로 충분하다.
주의점은 상태 정합성 목표 달성 잠금, 결제, 환불, 통합구매는 단순 CRUD보다 엄격한 상태 전이가 필요하다.
다음 단계에서 결제를 붙이면 Contribution, 결제 이벤트, 환불, 목표 달성 락, 구매 주문 상태를 트랜잭션 단위로 설계해야 한다.

권장 보강 순서

  1. 웹/Tauri 메뉴 데이터 분리 설계 확정: web_platform_menus, tauri_admin_menus 또는 menuType.
  2. 시설과 후원 물품 API의 공개/관리자 권한 경계를 명확히 한다.
  3. 후원 물품 상태 enum과 상태 전이 규칙을 테스트로 고정한다.
  4. 운영 전 DB migration 도구를 도입한다.
  5. 결제 전 단계에서는 사전등록 또는 더미 Contribution으로 흐름을 먼저 검증한다.
