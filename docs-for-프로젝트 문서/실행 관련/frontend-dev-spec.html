프론트엔드 개발 스펙 조사 및 적절성

DonationPlatform 웹 프론트는 후원자가 시설과 후원 물품을 탐색하고, 관리자가 기본 운영 메뉴를 사용할 수 있는 Next.js 앱이다. 현재 구조는 FSD에 가깝게 구성되어 있으며, shadcn/ui 스타일을 적용하기 좋은 상태다.

기본 스펙

항목 현재 스펙 적절성 판단
프로젝트 donation-platform-front 웹 플랫폼 전용 앱으로 유지하는 것이 맞다. 관리자 반복 업무는 Tauri로 분리하는 방향이 좋다.
프레임워크 Next.js 16.1.4, App Router 라우트 기반 페이지 구성에 적합하다. 최신 버전이라 생태계 호환성은 확인하며 가야 한다.
React React 19.2.3 최신 조합이다. 라이브러리 호환성 문제가 생기면 버전 고정이 중요하다.
언어 TypeScript 5 API 타입, 폼 값, 권한 모델을 다루기에 적절하다.
스타일 Tailwind v4, shadcn/ui 설정, Radix, CVA, lucide-react 모던 시크 운영형 UI를 빠르게 만들기에 적합하다.
서버 상태 TanStack React Query 5 API 캐시, refetch, mutation 관리에 적절하다.
폼/검증 react-hook-form, zod, hookform resolvers 관리자 폼과 후원 신청 폼 모두에 적합하다.
API 클라이언트 axios, token refresh interceptor JWT 인증 흐름에 적합하다. SSR fetch와 혼용할 때 기준을 정해야 한다.

shadcn/ui 설정 평가

항목 현재 설정 판단
components.json 존재함, schema https://ui.shadcn.com/schema.json shadcn/ui 기반 프로젝트로 보는 것이 맞다.
style new-york 모던하고 밀도 있는 UI에 적합하다.
baseColor neutral 모던 시크 컨셉과 잘 맞는다.
ui alias @/shared/ui FSD에서 shared primitive를 두기 좋은 설정이다.
iconLibrary lucide 현재 Header, 메뉴 아이콘과도 방향이 맞다.

상태 관리와 인증 흐름

  • QueryProvider에서 React Query를 전역 제공한다.
  • 기본 query staleTime은 30초, window focus refetch는 꺼져 있다.
  • AuthInitializer가 local storage 기반 인증 상태를 복구한다.
  • shared/api/axios.ts에서 access token 자동 첨부와 refresh token 재발급 큐를 처리한다.
  • 401이 발생하면 refresh를 시도하고, 실패 시 auth:logout 이벤트로 로그인 화면 이동을 유도한다.
MVP 기준으로 충분히 좋은 구조다. 다만 후원 공개 페이지는 비로그인 접근이 가능해야 하므로, 페이지별 인증 가드를 명확히 구분해야 한다.

MVP 화면 관점 적절성

후원자 웹 /donate, /my-donations 중심으로 가는 방향이 적절하다.
관리자 기능 웹에도 관리 화면이 있으나, 반복 운영 업무는 Tauri 앱으로 옮기는 것이 자연스럽다.
DB 메뉴 웹 사이드바를 DB 메뉴로 출력하는 현재 방향은 유지 가능하다.
웹은 후원 플랫폼 역할이므로 시설 CRUD, 권한 관리, 메뉴 관리 같은 운영 메뉴가 후원자 화면에 과하게 노출되지 않도록 메뉴 데이터와 권한 필터를 분리해야 한다.

현 스택의 장점과 리스크

구분 내용 판단
장점 Next App Router, TypeScript, React Query, shadcn/ui 조합 CRUD, 대시보드, 후원 목록 UI를 빠르게 만들기 좋다.
장점 FSD에 가까운 디렉터리 구조 도메인 확장 시 파일 위치 판단이 쉽다.
리스크 Next 16, React 19처럼 최신 버전 사용 일부 라이브러리 호환성 문제는 빌드에서 조기 확인해야 한다.
리스크 static export 설정 서버 액션, 동적 서버 렌더링이 필요한 기능은 쓰기 어렵다. 현재처럼 클라이언트 API 호출 중심이면 문제 없다.
리스크 웹과 관리자 기능 혼재 웹은 후원자 중심, Tauri는 관리자 중심으로 메뉴와 라우트를 점진 분리해야 한다.

적절성 결론

현재 프론트 스택은 1차 MVP에 적절하다. 특히 shadcn/ui + Tailwind + FSD 구조는 후원자 화면과 관리자성 화면을 모두 빠르게 만들 수 있다. 다만 웹의 역할은 후원 플랫폼으로 좁히고, 운영자 반복 업무는 Tauri 관리자 앱으로 분리하는 쪽이 장기적으로 더 깔끔하다.

권장 보강 순서

  1. 웹 메뉴를 후원자 중심으로 정리한다: 대시보드, 후원하기, 내 후원 내역.
  2. facility, donation-item, contribution entity 레이어를 추가한다.
  3. /donate에 시설 목록과 물품 미리보기 UI를 실제 컴포넌트로 구현한다.
  4. 관리자 전용 CRUD는 Tauri 앱으로 옮기고 웹에서는 노출을 줄인다.
  5. 공통 UI primitive를 shared/ui에 shadcn 스타일로 정리한다.
