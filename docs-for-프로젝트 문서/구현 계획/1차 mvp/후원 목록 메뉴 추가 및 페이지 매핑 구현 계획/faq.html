DonationPlatform Menu FAQ 시설 목록 · 후원 목록

후원 목록 메뉴 추가 및 페이지 매핑 FAQ

이 문서는 시설 목록, 후원 물품, 후원 목록 같은 신규 메뉴를 어디에 저장하고, 어떤 화면에서 출력하며, 웹 메뉴와 Tauri 관리자 앱 메뉴를 어떻게 분리할지 정리한다.

요약

현재 시스템은 기존 menus 테이블에 메뉴 데이터를 추가하고, 웹/타우리 클라이언트가 /api/menus 응답을 받아 페이지에 매핑하는 구조다. 따라서 지금 당장 시설 목록이나 후원 목록 메뉴를 추가하는 것은 가능하다.

단기 구현은 기존 menus 테이블을 그대로 써도 된다. 다만 웹 메뉴와 관리자 앱 메뉴가 계속 다르게 진화할 가능성이 높으므로, 중장기적으로는 web_menus와 admin_app_menus를 분리하는 편이 더 단순하고 안전하다.

1. 메뉴는 어느 테이블에 저장하나?

현재 메뉴 데이터는 서버의 Menu 엔티티가 매핑하는 menus 테이블에 저장한다. 이 테이블은 웹 좌측 사이드바 메뉴와 Tauri 관리자 앱 사이드바 메뉴 출력을 위해 사용하던 메뉴 테이블이다.

컬럼 의미
code 메뉴 식별 코드. 예: ADMIN_FACILITIES, ADMIN_CONTRIBUTIONS
parent_id 상위 메뉴. 예: ADMIN_DONATION 아래에 시설 목록을 둔다.
label 화면에 표시할 메뉴명. 예: 시설 목록
path 웹 라우트 또는 앱 내부 매핑 경로. 예: /facilities
icon 클라이언트에서 사용할 아이콘명. 예: Building2
requiredRole 메뉴 노출에 필요한 역할. 관리자 앱 메뉴는 보통 ROLE_PLATFORM_ADMIN
visible 메뉴 노출 여부
displayOrder 같은 부모 메뉴 안에서의 출력 순서
현재 기본 메뉴는 MenuSeeder에서 서버 시작 시 upsert된다. 신규 메뉴도 먼저 MenuSeeder에 추가하는 방식이 가장 빠르다. 
new MenuDef(
    "ADMIN_FACILITIES",
    "ADMIN_DONATION",
    "시설 목록",
    "nav.facilities",
    "/facilities",
    "Building2",
    RoleSeeder.ROLE_PLATFORM_ADMIN,
    0
)

2. 출력하는 페이지는 어디인가?

현재 메뉴 출력 위치는 웹과 Tauri 관리자 앱이 다르다. 둘 다 서버의 /api/menus를 호출하지만, 렌더링 위치와 페이지 매핑 방식은 다르다.

앱 메뉴 출력 위치 페이지 매핑 위치
웹 donation-platform-front/src/widgets/header/ui/Header.tsx 내부 Sidebar 컴포넌트 donation-platform-front/src/app/{path}/page.tsx
Tauri 관리자 앱 donation-admin-tauri/src/widgets/app-shell/ui/AppSidebar.tsx donation-admin-tauri/src/app/App.tsx 또는 별도 workspace 컴포넌트

웹 기준

웹은 Next.js App Router를 사용하므로 메뉴의 path가 /facilities라면 아래 파일을 만들면 된다. 

donation-platform-front/src/app/facilities/page.tsx

Tauri 관리자 앱 기준

Tauri 앱은 현재 브라우저 라우팅이 아니라 activeMenu 상태로 화면을 바꾸는 구조다. 따라서 ADMIN_FACILITIES 메뉴를 클릭했을 때 표시할 컴포넌트를 AdminWorkspace 또는 별도 라우팅/워크스페이스 매핑에 연결해야 한다. 

if (activeMenu === "ADMIN_FACILITIES") {
    return <FacilitiesPage />;
}

3. 웹에 출력하는 메뉴와 Tauri에 출력하는 메뉴는 어떻게 구분할 것인가?

현재는 하나의 menus 테이블을 같이 쓰고, 클라이언트에서 코드 규칙으로 필터링한다.

웹 필터 웹 사이드바를 렌더링하는 Header.tsx 내부 Sidebar 흐름에서 ADMIN, ADMIN_* 메뉴를 숨긴다.
Tauri 필터 navigation.ts의 isAdminAppMenu에서 DASHBOARD, ADMIN, ADMIN_* 메뉴만 표시한다. 
// 웹
function isWebMenu(menu) {
    if (menu.code === "DASHBOARD") return true;
    if (menu.code === "ADMIN" || menu.code.startsWith("ADMIN_")) return false;
    return true;
}

// Tauri 관리자 앱
function isAdminAppMenu(menu) {
    return menu.code === "DASHBOARD"
        || menu.code === "ADMIN"
        || menu.code.startsWith("ADMIN_");
}
이 방식은 지금 당장 구현하기는 빠르지만, 메뉴 코드 네이밍 규칙에 의존한다. 메뉴가 늘어나고 웹/관리앱의 정보 구조가 더 달라지면 관리가 지저분해질 수 있다.

테이블을 나누는 게 더 낫지 않나?

결론부터 말하면, 중장기적으로는 나누는 것이 더 속 편하다. 웹은 후원자가 보는 서비스 메뉴이고, Tauri는 플랫폼 운영자가 쓰는 관리자 메뉴다. 메뉴 성격, 권한, 화면 밀도, 배포 주기, 노출 정책이 다르다.

추천 방향은 web_menus와 admin_app_menus 분리다. 컬럼 구성은 거의 같아도 된다. 같은 테이블에 surface 컬럼을 추가하는 방법도 있지만, 앱 성격이 이미 분리되어 있으므로 별도 테이블이 더 명확하다.

분리안

테이블 용도 예시 메뉴
web_menus 후원자/시설 사용자가 웹에서 보는 메뉴 홈, 후원 물품, 내 후원 내역, 프로필
admin_app_menus 플랫폼 관리자용 Tauri 앱 사이드 메뉴 시설 목록, 후원 물품 관리, 후원 목록, 사용자 관리

별도 테이블 장점

  • 웹 메뉴와 관리자 앱 메뉴가 섞이지 않는다.
  • 클라이언트에서 ADMIN_ prefix 같은 규칙 필터를 덜어낼 수 있다.
  • 웹 메뉴 관리 화면과 관리자 앱 메뉴 관리 화면을 분리하기 쉽다.
  • 나중에 권한 정책이 달라져도 영향 범위가 작다.

별도 테이블 단점

  • 엔티티, Repository, Controller, Seeder가 하나 더 생긴다.
  • 메뉴 관리 UI도 분리하거나 공통 컴포넌트를 만들어야 한다.
  • 초기 구현량은 단일 테이블보다 조금 늘어난다.
지금 MVP 속도를 우선하면 기존 menus 테이블에 ADMIN_FACILITIES를 추가한다. 구조를 오래 가져갈 생각이면 다음 단계에서 admin_app_menus를 새로 만들고 Tauri 앱만 /api/admin-app-menus를 보게 바꾸는 것이 좋다.
