Step 1 후원하기 시설 단위 물품 후원

후원하기 페이지 기본 구현 계획

기획 문서 기준으로 1차 MVP의 핵심은 시설이 승인한 공용 필요 물품을 후원자가 품목 단위로 채우는 흐름이다. 따라서 웹의 후원하기 페이지는 시설 목록을 보여주되, 시설 탐색 자체보다 시설별 후원 가능 물품으로 빠르게 이동하고 후원 행동으로 이어지게 설계한다.

참고한 기획 방향

  • 1차 파일럿은 개별 아동 후원이 아니라 시설 단위 공용 물품 후원 모델로 시작한다.
  • 시설은 공용 필요 물품 5~10개를 제공하고, 운영팀은 이를 후원자용 카드/모집글로 전환한다.
  • 후원자는 개별 아동 사연 없이도 계절성 필요 물품에 참여할 수 있어야 한다.
  • 후원금은 시설 또는 운영법인에 귀속되고, 플랫폼은 후원자별 기록과 진행 상태를 관리한다.
  • 목표 금액이 채워지면 추가 결제를 막고, 주문·배송·수령·증빙 상태를 추적한다.
페이지의 중심 질문은 “어떤 시설이 있나?”가 아니라 “지금 내가 어떤 물품을 후원할 수 있나?”여야 한다.

추천 사용자 흐름

후원하기
  → 시설 목록/추천 시설 확인
  → 시설 카드 클릭
  → 해당 시설의 후원 가능 물품 목록
  → 물품 상세
  → 후원 금액 선택
  → 결제/후원 신청
  → 내 후원 내역에서 진행 상태 확인

첫 구현에서는 결제까지 바로 만들지 않고, 시설 목록 → 시설별 물품 목록 → 물품 상세까지만 페이지 구조를 잡아도 충분하다. 결제는 상태/정산/환불 정책이 붙으므로 다음 단계로 분리한다.

어떤 페이지부터 구현할 것인가?

순서 페이지 목적 초기 구현 범위
1 /donate 후원자가 후원 가능한 시설과 물품을 발견하는 진입점 시설 카드, 대표 물품 2~3개 미리보기, 진행률, 후원하기 CTA
2 /donate/facilities/{facilityId} 특정 시설의 후원 가능 물품 목록 확인 시설 소개, 공개 범위, 물품 카드 목록, 목표 금액/수량 진행률
3 /donate/items/{itemId} 물품 상세와 후원 참여 물품 설명, 필요 이유, 목표/달성 금액, 후원 금액 선택 UI
4 /my-donations 후원자가 본인 후원 진행 상태 확인 후원금액, 결제상태, 목표달성, 주문/배송/수령 상태
Tauri 관리자 앱의 시설 CRUD가 완성되기 전에는 샘플 데이터로 화면 구조를 먼저 잡고, 이후 Facility, DonationItem, Contribution API에 연결한다.

/donate 첫 화면 구성

상단 요약 진행 중인 후원 시설 수, 후원 가능 물품 수, 목표 달성 임박 물품 수를 보여준다.
필터 지역, 시설 유형, 물품 카테고리, 마감 임박, 목표 달성 임박 필터를 제공한다.
시설 카드 시설명 또는 공개명, 지역, 공개 가능 범위, 필요한 물품 개수, 대표 물품 미리보기를 보여준다.
물품 미리보기 시설 카드 안에 목표 금액/달성률이 보이는 물품 2~3개를 노출해 바로 후원 가능성을 전달한다.

Claude 목업 교차 비교 보강

후원하기 페이지 기본 with claude.html은 실제 화면 목업에 가깝고, 이 문서는 구현 순서와 데이터 연결 계획에 가깝다. 두 문서를 비교하면 아래 항목은 1차 구현 기준에 명시해두는 편이 좋다.

항목 Claude 목업에서 확인한 장점 구현 계획에 반영할 내용
탐색 구조 시설 목록과 시설 상세를 같은 후원 흐름 안에서 자연스럽게 전환한다. /donate는 시설 목록, /donate/facilities/[facilityId]는 시설별 물품 목록으로 분리하되 UX는 한 흐름처럼 보이게 한다.
검색/필터 시설명·지역 검색, 전체/모집중/지역/시설유형 칩이 있다. 초기에는 클라이언트 샘플 데이터 필터로 구현하고, API 연결 후 쿼리 파라미터로 확장한다.
시설 카드 시설명, 지역, 유형, 자격확인, 준비물 수, 모인 금액, 목표, 달성률을 한 카드에서 보여준다. 시설 카드는 소개보다 후원 가능성과 진행률을 먼저 보여주는 구조로 만든다.
품목 상태 모집중, 목표달성, 통합구매 진행중, 배송중 상태에 따라 CTA가 바뀐다. 상태 enum을 화면에 먼저 반영하고, 목표 달성 시 후원하기 버튼은 비활성화한다.
신뢰 안내 시설 전용계좌, 목표 달성 잠금, 아동 정보 비노출, 증빙/영수증 구분을 별도 영역으로 설명한다. 목록 하단 또는 상세 하단에 짧은 신뢰 안내 블록을 배치한다. 단, 앱 기능 설명처럼 길게 쓰지 않는다.
후원 참여 실결제 전에는 더미 모달로 시설, 품목, 금액, 귀속 계좌를 확인시킨다. 1차 MVP에서는 결제 대신 후원 의사 등록 또는 사전등록 흐름으로 막고, 실제 결제는 다음 단계로 둔다.

화면 상태 정의

상태 보여줄 내용 필수 동작
시설 목록 검색창, 필터 칩, 시설 카드 그리드, 신뢰 안내 시설 카드 클릭 시 해당 시설 물품 목록으로 이동
시설별 물품 목록 시설 공개명, 지역/유형 배지, 시설 설명, 물품 카드 목록 뒤로가기 또는 breadcrumb로 시설 목록 복귀
후원 가능 품목 품목명, 구성/수량, 목표 금액, 모인 금액, 달성률, 후원하기 버튼 후원하기 클릭 시 후원 참여 확인 단계로 이동
잠긴 품목 목표달성, 통합구매 진행중, 배송중 등 상태 배지 추가 후원 버튼 비활성화
빈 목록 현재 공개된 후원 물품이 없다는 안내 필터 초기화 또는 대시보드 이동 제공
오류/로딩 데이터를 불러오는 중, 다시 시도 안내 재시도 버튼과 기본 skeleton 또는 빈 카드 제공

시설 상세의 역할

시설 상세는 시설 소개 페이지가 아니라 해당 시설의 후원 가능 물품 목록 페이지다. 아동 실명, 얼굴, 개인 사연, 직접 연락 정보는 노출하지 않는다.

  • 시설 공개명 또는 지역+시설 유형 표시
  • 시설 담당자가 승인한 공용 필요 물품 목록 표시
  • 각 물품의 목표 금액, 달성률, 남은 금액, 마감일 표시
  • 물품별 후원하기 CTA 제공
  • 목표달성, 통합구매 진행중, 배송중 상태에서는 CTA를 잠그고 상태 배지를 우선 표시
  • 물품 중심 수령 확인과 결과 리포트 예정 안내

데이터 모델 연결 순서

모델 역할 후원하기 화면에서의 사용
Facility 후원 물품이 연결되는 기준 시설 시설 카드, 시설 상세 헤더
DonationItem 시설이 승인한 공용 필요 물품 물품 카드, 목표 금액/수량, 상세 페이지
Contribution 후원자가 특정 물품에 참여한 기록 달성률 계산, 내 후원 내역
PurchaseOrder 목표 달성 후 통합 구매 처리 후원 이후 진행 상태

구현 순서

  1. /donate 페이지를 샘플 데이터 기반 시설 카드 + 검색/필터 + 진행률 구조로 구현한다.
  2. /donate/facilities/[facilityId] 페이지를 만들고 시설별 물품 목록, 상태 배지, CTA 잠금 규칙을 표시한다.
  3. /donate/items/[itemId] 페이지 또는 모달을 만들고 물품 상세/후원 참여 확인 UI를 배치한다.
  4. Tauri 관리자 앱에서 시설과 후원 물품 CRUD를 구현한다.
  5. 웹 샘플 데이터를 실제 Facility, DonationItem API로 교체한다.
  6. 후원 신청 또는 결제 전 단계 기록인 Contribution 생성 흐름을 붙인다.
  7. 결제, 목표 달성 잠금, 구매/배송/수령/증빙은 다음 단계로 분리한다.
실결제는 전문가 사전 검토와 자금·영수증·환불 정책이 정리되기 전까지 열지 않는다. 1차 화면 구현은 후원 흐름과 데이터 구조 검증에 집중한다.
