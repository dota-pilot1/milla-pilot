1차 MVP Facility CRUD Tauri Admin + Web

시설물 관리 기능 목록 CRUD 구현 계획

1차 MVP는 Tauri 관리자 앱에서 시설을 등록·관리하고, 웹에서는 후원자가 시설 목록을 조회하는 흐름으로 시작한다. 시설은 후원 물품과 후원 목록의 기준 데이터이므로 가장 먼저 구현한다.

결론

지금 바로 메뉴 한두 개를 추가하고 시설 CRUD를 구현하는 방향은 적절하다. 다만 웹과 Tauri 관리자 앱의 메뉴는 앞으로 계속 다르게 진화할 가능성이 높으므로, 기능 구현 전에 메뉴 테이블 분리를 먼저 하는 편이 더 깔끔하다.

추천 순서: tauri_admin_menus 신설 + 기존 메뉴를 web_platform_menus로 전환 → 메뉴 시딩 → 페이지 매핑 → 앱과 웹에서 테스트 → Facility CRUD 구현.

왜 시설부터인가?

  • 후원 물품은 어느 시설이 필요로 하는 물품인지 연결되어야 한다.
  • 후원 목록은 결국 특정 시설 또는 시설의 물품에 대한 후원 기록이다.
  • 시설 관리자가 로그인해서 관리할 수 있는 기준 데이터가 필요하다.
  • 웹에서 후원자가 볼 첫 화면도 시설 또는 시설별 후원 물품 목록이 자연스럽다.

메뉴 테이블 분리 여부

현재는 menus 테이블 하나를 웹과 Tauri 앱이 같이 사용한다. 클라이언트에서 ADMIN_ prefix로 필터링하면 당장 구현은 가능하지만, 장기적으로는 관리 포인트가 흐려진다.

선택지 장점 단점 판단
기존 menus 유지 가장 빠르게 메뉴 추가 가능 웹/Tauri 메뉴가 한 테이블에 섞이고 클라이언트 필터에 의존 임시 MVP에는 가능
surface 컬럼 추가 테이블은 하나만 유지하면서 구분 가능 웹/앱 메뉴 관리 화면과 정책이 계속 한 테이블에 묶임 중간안
web_platform_menus, tauri_admin_menus 분리 웹 메뉴와 관리자 앱 메뉴의 책임이 명확함 엔티티/API/시더가 하나 더 필요 추천
최종 명칭은 web_platform_menus와 tauri_admin_menus로 둔다. 기존 menus는 웹 플랫폼 메뉴 성격이므로 web_platform_menus로 전환하고, Tauri 관리자 앱은 별도 tauri_admin_menus 테이블과 API를 사용한다.

메뉴 분리 후 API 명칭

테이블명을 그대로 드러내면 클라이언트 입장에서 어느 메뉴를 호출해야 하는지 명확하다. 기존 /api/menus는 모호하므로 장기적으로 deprecated 처리한다.

테이블 API 사용 클라이언트
web_platform_menus GET /api/web-platform-menus 웹 후원 플랫폼
tauri_admin_menus GET /api/tauri-admin-menus Tauri 관리자 앱
기존 웹 코드가 /api/menus를 호출하고 있으므로, 전환 시점에는 /api/menus를 잠시 유지하거나 웹 클라이언트를 동시에 /api/web-platform-menus로 바꿔야 한다.

웹에는 무엇을 추가할 것인가?

웹은 운영자 도구가 아니라 후원자가 보는 서비스 화면이다. 따라서 관리자 표현인 시설 관리보다 후원 시설 또는 시설 둘러보기가 적절하다.

1차 추천 메뉴 후원 시설 / /facilities
후원자가 공개된 시설 목록을 조회한다.
2차 확장 메뉴 후원 물품 / /donation-items
시설별 필요 물품을 조회하고 후원한다.
웹에서 시설 등록/수정은 넣지 않는다. 시설 등록은 Tauri 관리자 앱 또는 시설 관리자 전용 관리 화면에서 처리하고, 웹은 공개 조회와 후원 흐름에 집중한다.

시설 관리자 계정과 시설을 연결하는 기능은 1차 CRUD 이후에 붙인다. 처음에는 플랫폼 관리자가 시설을 등록하고, 후원자가 웹에서 공개 시설을 조회하는 흐름만 구현한다.

구현 순서

  1. 메뉴 반영 및 데이터 시딩 설정: tauri_admin_menus, web_platform_menus 기준으로 엔티티/API/시더를 정리하고 ADMIN_FACILITIES, WEB_FACILITIES 메뉴를 넣는다.
  2. 추가한 메뉴들에 페이지 설정: Tauri는 ADMIN_FACILITIES를 시설 관리 화면에 매핑하고, 웹은 WEB_FACILITIES를 /facilities 공개 조회 페이지에 매핑한다.
  3. 앱과 웹에서 테스트: Tauri 앱 사이드바에 시설 관리가 보이는지, 웹 사이드바에 후원 시설이 보이는지, 각 메뉴 클릭 시 의도한 페이지가 열리는지 확인한다.
  4. 그다음 서버 Facility 엔티티와 CRUD API를 구현한다.
  5. Tauri 앱 시설 목록/등록/수정 화면을 실제 API에 연결한다.
  6. 웹 공개 시설 목록을 실제 API에 연결한다.
  7. 후원 물품 등록 기능으로 연결한다.
