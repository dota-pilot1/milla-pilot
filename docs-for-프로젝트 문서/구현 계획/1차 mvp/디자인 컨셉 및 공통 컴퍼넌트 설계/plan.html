1차 MVP Design System shadcn/ui FSD

디자인 컨셉 및 공통 컴포넌트 설계

DonationPlatform의 1차 MVP 디자인은 shadcn/ui 스타일을 기준으로 한 모던 시크 톤으로 통일한다. 웹은 후원자가 빠르게 후원 대상을 탐색하고 참여하는 화면, Tauri 앱은 관리자가 반복 업무를 처리하는 운영 콘솔로 구분한다.

디자인 컨셉

Modern Chic 과한 장식보다 선명한 정보 구조, 넓은 여백, 얇은 border, 낮은 채도의 색을 사용한다.
shadcn/ui Base Radix 접근성, Tailwind utility, CVA variant, lucide icon을 기본 컴포넌트 규칙으로 삼는다.
Donation First 웹 화면은 관리 용어보다 후원자 행동 언어를 우선한다. 예: 관리 → 보기/후원하기/내역.
결론: 디자인 키워드는 shadcn/ui 기반의 모던 시크 운영형 UI다. 화려한 랜딩 페이지보다 실제 업무와 후원 흐름이 잘 보이는 화면을 우선한다.

시각 규칙

영역 규칙 비고
색상 기본은 neutral/white/black 계열, 강조는 green 또는 blue를 제한적으로 사용한다. 기부/후원 CTA는 green, 시스템 정보나 링크는 blue.
라운드 버튼과 입력은 8~10px, 큰 패널은 12~16px 정도로 제한한다. 과한 pill/card 남발 금지.
타이포 페이지 제목은 굵고 명확하게, 카드 내부 제목은 작고 단단하게 사용한다. viewport 기반 font-size scaling 금지.
아이콘 lucide-react를 기본으로 사용한다. 직접 SVG를 그리기보다 기존 아이콘 우선.
상태 표현 badge, progress, disabled button으로 상태를 명확히 분리한다. 모집중, 목표달성, 구매중, 배송중, 수령완료.
다크 모드 1차 MVP 문서와 기본 화면은 light mode 우선. 다크 모드는 추후 테마 작업에서 분리한다.

공통 컴포넌트 분류

분류 컴포넌트 예시 FSD 위치 판단 기준
순수 UI Primitive Button, Input, Textarea, Select, Dialog, Badge, Progress, Tabs src/shared/ui 도메인 단어 없이 어디서나 재사용 가능하면 shared.
레이아웃/프레임 AppShell, Sidebar, TopBar, PageHeader, SectionPanel src/widgets/*/ui 여러 shared UI를 조합해 화면 뼈대를 만들면 widget.
도메인 카드 FacilityCard, DonationItemCard, ContributionStatusCard src/entities/{domain}/ui 특정 엔티티 데이터를 보여주는 재사용 가능한 표시 컴포넌트.
사용자 행동 DonateButton, FacilitySearchFilter, CreateFacilityForm, DonationAmountForm src/features/{feature}/ui 클릭/제출/API 호출 등 사용자의 행동 단위가 있으면 feature.
페이지 /donate, /donate/facilities/[id], /dashboard src/app/*/page.tsx 라우팅, 데이터 조합, page-level composition만 담당.

우선 만들 공통 컴포넌트

우선순위 컴포넌트 사용처 비고
1 Button, Input, Badge, Progress 웹 후원하기, Tauri 시설 관리 shadcn 스타일 primitive 먼저 정리.
2 PageHeader, SectionPanel, EmptyState 대부분의 페이지 페이지마다 반복되는 제목/설명/빈 상태 통일.
3 FacilityCard, DonationItemCard 웹 후원하기 후원자용 도메인 표시 컴포넌트.
4 DataTable, FormShell, StatusBadge Tauri 관리자 앱 시설/물품 CRUD에 바로 필요.

컴포넌트 작성 규칙

  1. 공통 primitive는 도메인 단어를 넣지 않는다. 예: Button, Badge.
  2. 도메인 표시 컴포넌트는 entities에 둔다. 예: FacilityCard.
  3. 등록, 수정, 삭제, 검색처럼 행동이 있는 UI는 features에 둔다.
  4. 페이지 전체 레이아웃은 widgets에 둔다. 예: sidebar, app shell.
  5. 라우트 파일은 조립만 담당하고, UI 로직을 길게 넣지 않는다.
  6. variant가 필요한 컴포넌트는 class-variance-authority를 사용한다.
  7. 아이콘은 lucide-react를 우선 사용한다.

1차 적용 순서

  1. 웹과 Tauri의 shared/ui primitive를 shadcn 스타일로 정리한다.
  2. 웹 /donate 페이지에 FacilityCard, DonationItemCard, FacilityFilterBar를 적용한다.
  3. Tauri 시설 관리 화면에 DataTable, FormShell, StatusBadge를 적용한다.
  4. 중복 스타일이 생기면 shared primitive로 내리고, 도메인 규칙은 entities/features에 남긴다.
공통 컴포넌트를 처음부터 과하게 만들지 않는다. 실제로 두 번 이상 반복되는 UI만 shared로 내리고, 도메인 맥락이 강한 컴포넌트는 entities 또는 features에 남긴다.
